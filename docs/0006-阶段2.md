# DeployLab 阶段二:后端 Docker 化 · 执行方案

> 对应 `0001-总计划.md` 第九章「第 2 阶段:后端 Docker 化」。
> 一句话目标:**把 Express 后端打成 Docker 镜像,在 hp-linux 上以容器形式跑起来,并连上第 1 阶段的 MySQL 容器**。
> 这一阶段最重要的一个认知,也是总计划反复强调的:**`DB_HOST` 不能写 `localhost`,要写 MySQL 容器的名字**(原因见第八章)。

---

## 〇、阶段定位与边界

### 0.1 这一阶段的部署形态

第 1 阶段是「Win11 跑前后端 + hp-linux 只放 MySQL」。第 2 阶段把**后端也搬上 hp-linux、塞进容器**:

```txt
Win11 开发机                              hp-linux server (Ubuntu 24.04)
┌───────────────────────┐                ┌──────────────────────────────────────┐
│ Vue dev server  :5173 │                │  Docker 用户网络:deploy-lab-net       │
│  └ Vite proxy /api    │ ─Tailscale──▶ │  ┌───────────────┐    ┌──────────────┐ │
│     → 100.98.102.107  │  100.98.102.107│  │ deploy-server │───▶│ deploy-mysql │ │
│        :3000          │   仅这一段跨机 │  │  Express:3000 │    │  MySQL:3306  │ │
└───────────────────────┘                │  └───────────────┘    └──────────────┘ │
                                         │   -p 3000:3000          -p 3306:3306    │
                                         └──────────────────────────────────────┘
```

请求流向(注意跨机器的只有最前面一段):

```txt
浏览器 :5173 ─(Vite proxy /api)─▶ deploy-server:3000 ─(deploy-lab-net 容器 DNS)─▶ deploy-mysql:3306
   │ Win11                            │ hp-linux 容器                                      │ 同机内网
   └── 跨机器这段才走 Tailscale 100.98.102.107:3000 ──┘                                 └── server→MySQL 不再走 Tailscale
```

### 0.2 与原方案 / 上一阶段的关键差异

| 维度 | 第 1 阶段 | 第 2 阶段(本方案) |
|------|-----------|--------------------|
| Express 跑在哪 | Win11 本地 node 进程 | **hp-linux 上的容器 `deploy-server`** |
| `DB_HOST` 写什么 | `100.98.102.107`(Tailscale) | **`deploy-mysql`**(容器名,走容器网络) |
| server→MySQL 走什么 | Tailscale 跨机 | **同机的 Docker 内部网络,不再依赖 Tailscale** |
| 前端怎么找后端 | Vite proxy → `localhost:3000` | Vite proxy → `100.98.102.107:3000` |

这样做的好处:**后端一旦容器化,它和 MySQL 就在「同一个 Docker 网络房间」里**,互相用名字直连,既快又不依赖 Tailscale——这正是后面 Compose 编排、Nginx 反代的基石。

代价:前端(Vite)还在 Win11,要跨机器找容器里的后端,所以 hp-linux 上后端容器的 3000 端口要临时对外(见第六章,⚠️ 第 5 阶段收紧)。

### 0.3 这一阶段「不做什么」(防止跑偏)

| 不做的事 | 留到哪个阶段 |
|---------|-------------|
| `docker-compose.yml` 多容器编排 | 第 3 阶段 |
| Redis、`/api/stats` 访问计数 | 第 4 阶段 |
| Nginx 托管前端 + 反代 | 第 5 阶段 |
| 前端容器化(dist + Nginx 镜像) | 第 5 阶段 |
| 只对外暴露 80、撤掉 3000/3306 外部映射 | 第 5 阶段 |

> 第 2 阶段就是「写一个 Dockerfile、build、run、连上 MySQL」四件事。心智越简单越好。

### 0.4 ⚠️ 先说清楚:`DB_HOST` 到底写 `mysql` 还是 `deploy-mysql`?

总计划第 2 阶段写的是「应该写 `mysql`」,那是因为总计划设想的最终形态是 **Compose**,compose 里 MySQL 的**服务名**叫 `mysql`,所以 `DB_HOST=mysql`。

但第 1 阶段实际执行时,MySQL 是用 `docker run` 单独起的,容器名叫 **`deploy-mysql`**(不是 `mysql`)。本阶段还**没到 Compose**,所以:

```txt
本阶段(第 2 阶段,无 compose):DB_HOST = deploy-mysql   ← 用现有容器的真实名字
下一阶段(第 3 阶段,有 compose):DB_HOST = mysql        ← compose 把 MySQL 服务命名为 mysql
```

**原理完全一样:容器名 / 服务名 = Docker 内部网络的 DNS 名**。别被 `mysql` vs `deploy-mysql` 这两个字符串绕住——记住「写对方的名字,别写 localhost」就够了。下面统一用 `deploy-mysql`(因为它是这台机器上 MySQL 容器的真实名字)。

## 二、写 Dockerfile + .dockerignore(Win11 本地)

> 这两个文件在 Win11 写好,第三章再传到 hp-linux 去 build。镜像在 hp-linux 上 build,因为后端容器要在 hp-linux 上跑(和 MySQL 同机)。

### 2.1 `server/Dockerfile`

新建 `server/Dockerfile`,内容:

```dockerfile
# 基础镜像:Node 20 LTS 的 Alpine 版,体积小、够用
FROM node:20-alpine

# 容器内工作目录
WORKDIR /app

# 关键:先只拷「依赖描述文件」,再装依赖
# 这样只要依赖没变,这一层就能命中缓存,改业务代码 rebuild 会快很多
COPY package.json package-lock.json ./
# --omit=dev:nodemon 这类开发依赖不打进生产镜像
RUN npm ci --omit=dev

# 依赖装好后再拷业务代码(改动最频繁的放最后)
COPY src ./src

# 文档性声明:容器内监听 3000(不会自动对外发布,对外发布靠 docker run 的 -p)
EXPOSE 3000

# 生产环境用 node 直接跑,不用 nodemon
CMD ["node", "src/app.js"]
```

逐行要理解的概念:

```txt
FROM node:20-alpine   镜像是分层缓存;基础层最稳,放最上面
WORKDIR /app          后续命令的「当前目录」一律是 /app
COPY 依赖文件→npm ci  分两步拷贝是 Docker 最佳实践:依赖层可缓存
--omit=dev            等同旧版 --only=production;省体积、减攻击面
EXPOSE 3000           只是声明;真正发布端口是 docker run 的 -p 3000:3000
CMD                   容器启动命令;写成数组形式(exec 形式)能正确收到信号
```

> 注意:Dockerfile 里**没有** `COPY .env`。环境变量(含数据库密码)绝不打进镜像,第六章用 `docker run -e` 在运行时注入。

### 2.2 `server/.dockerignore`(很重要,别漏)

新建 `server/.dockerignore`:

```gitignore
# 本地依赖(Win11 装的,平台可能不对,容器里要重新 npm ci)
node_modules

# 环境变量(含密码!绝对不能打进镜像)
.env
.env.*

# 日志 / 编辑器 / 系统
*.log
.DS_Store
.vscode
.idea

# 版本管理
.git
.gitignore
```

为什么这么重要:

```txt
1) 不写 .dockerignore,COPY 会把 Win11 的 node_modules 一起拷进镜像
   → 那是 Windows 平台的依赖,跑在 Linux 容器里可能直接报错
   → 且镜像体积暴增。.dockerignore 把它挡掉,容器内靠 npm ci 重装
2) 不挡 .env,密码会被烘焙(bake)进镜像层
   → 镜像一旦 push/分享就泄密。环境变量必须运行时注入(第六章)
```

### 2.3 环境变量「不打进镜像」原则(本阶段的核心纪律)

| 项 | 错误做法 | 正确做法 |
|----|---------|---------|
| 数据库密码 | 写进 Dockerfile / COPY .env | `docker run -e DB_PASSWORD=...` |
| `DB_HOST` | 在镜像里硬编码 | 运行时 `-e DB_HOST=deploy-mysql` |
| 同一份镜像 | 不同环境各 build 一次 | **一份镜像,用不同 `-e` 跑到不同环境** |

> 这条纪律贯穿后面所有阶段,务必第 2 阶段就建立起来。

## 四、建网络 + 接 MySQL(容器名解析的前提)

> ⭐ 这是本阶段的「隐形关键步」:**只有「用户自定义网络」才支持「容器名 = DNS 名」**。默认的 `bridge` 网络不解析容器名。所以要让 `DB_HOST=deploy-mysql` 能用,得先把两个容器放到同一个用户自定义网络里。

### 4.1 创建用户自定义网络

```bash
# 在 hp-linux 上
docker network create deploy-lab-net
docker network ls | grep deploy-lab-net     # 应能看到 deploy-lab-net
```

### 4.2 把第 1 阶段的 MySQL 容器接进来(不重建,数据不动)

```bash
# deploy-mysql 原本在默认 bridge 上,现在再给它接一条 deploy-lab-net
docker network connect deploy-lab-net deploy-mysql
```

> `connect` 不会重启容器、不丢数据,只是让 `deploy-mysql` **同时**挂在 `deploy-lab-net` 上(一个容器可属于多个网络)。它原先的 `-p 3306:3306` 外部映射照常保留。

### 4.3 验证网络成员

```bash
docker network inspect deploy-lab-net
# 看 "Containers" 段:应能看到 deploy-mysql(等下加上 deploy-server 就有两个)
```

成功标志:`deploy-mysql` 出现在 `deploy-lab-net` 的 Containers 列表里。

## 六、运行 server 容器(hp-linux)

### 6.1 docker run(用 -e 注入环境变量)

```bash
docker run -d \
  --name deploy-server \
  --network deploy-lab-net \
  --restart unless-stopped \
  -p 3000:3000 \
  -e PORT=3000 \
  -e DB_HOST=deploy-mysql \
  -e DB_PORT=3306 \
  -e DB_USER=root \
  -e DB_PASSWORD=123456 \
  -e DB_NAME=deploy_lab \
  -e APP_VERSION=v1.0.0 \
  deploy-lab-server:v1
```

参数逐条理解(本阶段必须看懂):

```txt
-d                       后台跑
--name deploy-server     容器名;后面 docker logs/exec 都用它
--network deploy-lab-net ⭐ 关键:挂到第四章建的网络,才能用名字解析到 deploy-mysql
-p 3000:3000             把容器 3000 发布到 hp-linux 的 0.0.0.0:3000
                         ⚠️ 临时对外,方便 Win11 的 Vite 跨机访问;第 5 阶段撤掉
-e ...                   运行时注入环境变量(密码不进镜像)
                         ⭐ DB_HOST=deploy-mysql 就是总计划说的「别写 localhost」
--restart unless-stopped 宿主机重启/容器异常退出后自动拉起
deploy-lab-server:v1     用第五章 build 出来的镜像
```

> `-e` 太多时,可改成 `--env-file`:在 hp-linux 建个 `~/code/person/deploy-lab/server/docker.env`(内容就是上面这些 `KEY=VALUE`,**别提交到 git**),然后 `docker run ... --env-file docker.env ...`。效果一样,更适合变量多时。

## 八、核心理解:为什么 `DB_HOST` 不能写 `localhost` ⭐

这是第 2 阶段最重要的一节,务必想透。

### 8.1 `localhost` 在容器里指谁?

每个容器都是**独立的网络空间**。容器里的 `localhost`(127.0.0.1)指的是**这个容器自己**。

```txt
deploy-server 容器里写 DB_HOST=localhost
  → 意思是「MySQL 就在 deploy-server 这个容器里」
  → 但 MySQL 根本不在它肚子里,在隔壁的 deploy-mysql 容器
  → 连接必然 ECONNREFUSED
```

类比:容器是各自独立的「房间」,`localhost` 是「我自己这间房」。你的 MySQL 在隔壁房间,你得喊**隔壁房间的名字(deploy-mysql)**,而不是喊「我自己」。

### 8.2 为什么第 1 阶段的 `100.98.102.107` 现在不该用?

第 1 阶段 Express 跑在 Win11 上,MySQL 在 hp-linux 上,**只能**借 Tailscale 跨过去,所以 `DB_HOST=100.98.102.107` 是对的。

现在后端和 MySQL **同在 hp-linux 的一台机器、同一个 Docker 网络里**,再绕出去走 Tailscale 就:① 多一次跨机网络跳转,慢;② 依赖 Tailscale 在线,Tailscale 一抖后端就连不上库。容器化后该走**容器内部网络**。

### 8.3 容器名怎么变成 IP 的?

```txt
1) 你把 deploy-server 和 deploy-mysql 都放进 deploy-lab-net(第四章)
2) Docker 给这个用户自定义网络内置了一个 DNS 服务
3) deploy-server 容器里发起到 deploy-mysql:3306 的连接时
   → Docker DNS 把 deploy-mysql 解析成它在该网络里的容器 IP(见 6.4 的 nslookup)
   → 连接直达,无需知道 IP 是多少(IP 是动态的,名字才稳定)
```

> ⚠️ 注意:**默认的 `bridge` 网络没有这个 DNS**。所以如果跳过第四章、让 server 容器停在默认 bridge 上,`deploy-mysql` 这个名字解析不了,`db` 会是 `down`。这就是第四章必须建 `deploy-lab-net` 的原因。

### 8.4 `DB_HOST` 在各阶段的演进(一图记住)

| 阶段 | `DB_HOST` | 为什么是这个值 |
|------|-----------|----------------|
| 第 1 阶段 | `100.98.102.107` | Express 在 Win11,只能借 Tailscale 连 hp-linux 的 MySQL |
| **第 2 阶段(本阶段)** | **`deploy-mysql`** | Express 容器与 MySQL 容器同在 deploy-lab-net,走容器 DNS |
| 第 3 阶段(Compose) | `mysql` | compose 把 MySQL **服务**命名为 mysql,服务名即 DNS 名 |
| 第 4 阶段(加 Redis) | 不变,Redis 同理写 `redis` | 同一个 compose 网络里,服务名解析 |

> 总计划写的「应该写 `mysql`」是**第 3 阶段及以后**的稳定值;本阶段用现有容器的真实名字 `deploy-mysql`,原理相同。

### 8.5 对照实验(可选,加深印象,也是第 7 阶段故障排查的预演)

故意造一个错,体会区别:

```bash
# A. 临时起一个「不挂 deploy-lab-net」的同镜像容器,看它能不能解析 deploy-mysql
docker run --rm -it --network bridge deploy-lab-server:v1 \
  sh -c "nslookup deploy-mysql || echo '解析失败:默认 bridge 没有容器名 DNS'"

# B. 同样这个容器,把 DB_HOST 故意写 localhost
docker run --rm -e DB_HOST=localhost -e DB_PORT=3306 -e DB_USER=root \
  -e DB_PASSWORD=123456 -e DB_NAME=deploy_lab --network bridge deploy-lab-server:v1
#   → /api/health 会返回 db:down,日志 ECONNREFUSED 127.0.0.1:3306
#   (Ctrl+C 退出;--rm 会自动清理)
```

看到 B 报 `ECONNREFUSED 127.0.0.1:3306`,就彻底懂了「localhost 指容器自己」这句话。

## 十、常见坑速查表

| 现象 | 大概率原因 | 怎么查 / 怎么修 |
|------|-----------|----------------|
| `/api/health` 返回 `db:down`、日志 `ECONNREFUSED 127.0.0.1:3306` | `DB_HOST` 写成了 `localhost` | 改 `-e DB_HOST=deploy-mysql` 后 `docker run` 重建容器 |
| `db:down`、日志 `getaddrinfo ENOTFOUND deploy-mysql` | server 容器没挂 `deploy-lab-net`,或 deploy-mysql 没 connect 进来 | `docker network inspect deploy-lab-net` 看两容器是否都在;补 `--network` / `docker network connect` |
| `db:down`、`ER_ACCESS_DENIED_ERROR` | 密码 / 库名错 | 核对 `-e` 的 `DB_PASSWORD`、`DB_NAME` 与 MySQL 容器一致(都是 123456 / deploy_lab) |
| build 时镜像塞进了 Win11 的 node_modules,跑起来报错 | 没写 `.dockerignore` | 加 `server/.dockerignore` 排除 node_modules,rebuild |
| 密码被打进镜像 / git | `COPY . .` 把 `.env` 带进去 | `.dockerignore` 排除 `.env`;env 一律 `-e` 注入 |
| `docker build` 报 `npm ci` 找不到 lock | 没传 `package-lock.json` | 按第三章把 `package-lock.json` scp 过去 |
| hp-linux build `npm ci` 很慢 / 超时 | 走国外 npm 源 | 见 5.2 换 npmmirror 镜像 |
| Win11 `curl 100.98.102.107:3000` 不通 | 3000 没绑 0.0.0.0 / 防火墙 / Tailscale 断 | `docker port deploy-server`(应是 0.0.0.0:3000);`tailscale status` |
| 改了 `vite.config.ts` 前端还是打到 `localhost:3000` | Vite 没重启 | Ctrl+C 重启 `npm run dev` |
| deploy-mysql 突然连不上 | 误把它从 deploy-lab-net 断开,或容器停了 | `docker network inspect deploy-lab-net`、`docker ps` |

### 12.2 给后面阶段埋的伏笔(心里有数即可)

```txt
第 3 阶段(Compose 编排):
  - 把 server + mysql 写进 docker-compose.yml
  - compose 自动建网络、自动服务名解析 → MySQL 服务名设为 mysql
  - DB_HOST 从 deploy-mysql 改回 mysql;复用 deploy_mysql_data volume,数据不丢
  - 撤掉本阶段手动的 deploy-lab-net / docker run,统一交给 compose up -d

第 4 阶段(加 Redis):
  - 同样在 compose 网络里加 redis 服务,server 用 DB 那套方式连 redis:6379

第 5 阶段(前端 + Nginx + 安全收紧):
  - 前端容器化(dist + Nginx),统一入口走 80
  - server 的 3000、MySQL 的 3306 外部映射全部撤掉,只走内部网络
  - 对外只剩 80
```

第 2 阶段把「后端容器化 + 容器间用名字互连」这条路打通。后面几个阶段,本质都是「把更多服务搬进同一个 Docker 网络、再用 compose 串起来」,业务代码基本不用动。
