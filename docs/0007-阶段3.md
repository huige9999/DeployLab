# DeployLab 阶段三:Compose 编排 server + mysql · 执行方案

> 对应 `0001-总计划.md` 第九章「第 3 阶段:Compose 编排 server + mysql」。
> 一句话目标:**用 `docker-compose.yml` 把 server 和 mysql 编进同一个 Compose 项目,一条 `docker compose up -d` 一起拉起,不再靠手动的 `docker run` + `docker network connect`**。
> 这一阶段最重要的变化:**`DB_HOST` 从第 2 阶段的 `deploy-mysql` 改成 Compose 服务名 `mysql`**——原理不变,只是「对方的名字」换成了 compose 里定义的服务名。

---

## 〇、阶段定位与边界

### 0.1 这一阶段的部署形态

第 2 阶段是「两个容器各自 `docker run`,再手动把它们挂进 `deploy-lab-net`」。第 3 阶段把这些**写进一份 compose 文件,交给 Compose 统一管**:

```txt
Win11 开发机                              hp-linux server (Ubuntu 24.04)
┌───────────────────────┐                ┌──────────────────────────────────────────┐
│ Vue dev server  :5173 │                │  Compose 项目:deploy-lab                  │
│  └ Vite proxy /api    │ ─Tailscale──▶ │  默认网络:deploy-lab_default(自动创建)   │
│     → 100.98.102.107  │  100.98.102.107│  ┌─────────────┐      ┌─────────────┐   │
│        :3000          │                │  │   server    │─────▶│    mysql    │   │
└───────────────────────┘                │  │ Express:3000│      │  MySQL:3306 │   │
                                         │  └─────────────┘      └─────────────┘   │
                                         │   -p 3000:3000          -p 3306:3306     │
                                         │   DB_HOST=mysql         volume 复用       │
                                         └──────────────────────────────────────────┘
```

请求流向(与第 2 阶段相比,**跨机段不变**,变的只是 hp-linux 上「怎么起容器」):

```txt
浏览器 :5173 ─(Vite proxy /api)─▶ server:3000 ─(Compose 网络 DNS)─▶ mysql:3306
   │ Win11                            │ hp-linux 容器                              │ 同 compose 网络
   └── 仍走 Tailscale 100.98.102.107:3000 ──┘                                  └── 服务名 mysql 解析
```

### 0.2 与第 2 阶段的关键差异

| 维度 | 第 2 阶段 | 第 3 阶段(本方案) |
|------|-----------|--------------------|
| 怎么起容器 | 两次 `docker run` + `docker network connect` | **`docker compose up -d` 一条命令** |
| 网络谁建 | 手动 `docker network create deploy-lab-net` | **Compose 自动建 `deploy-lab_default`** |
| `DB_HOST` 写什么 | `deploy-mysql`(旧容器名) | **`mysql`**(compose 服务名) |
| MySQL 容器名 | `deploy-mysql` | compose 管理(如 `deploy-lab-mysql-1`,不重要) |
| server 容器名 | `deploy-server` | compose 管理(如 `deploy-lab-server-1`) |
| 配置放哪 | 散落在 shell 历史 / `docker run` 参数里 | **集中在 `docker-compose.yml`** |
| 数据 volume | `deploy_mysql_data` | **复用同一个 `deploy_mysql_data`,数据不丢** |

这样做的好处:**编排、网络、环境变量、端口映射都进版本库**,后面加 Redis、加 Nginx 只是在 compose 里多写几段 service,不用记一长串 `docker run`。

代价:要从「手工容器」**迁移**到 compose 管理的容器——第 1、2 阶段起的 `deploy-mysql` / `deploy-server` 要先停掉再删容器(**volume 不能删**),本章会一步步写清楚。

### 0.3 这一阶段「不做什么」(防止跑偏)

| 不做的事 | 留到哪个阶段 |
|---------|-------------|
| Redis、`/api/stats` 访问计数 | 第 4 阶段 |
| 前端容器化 + Nginx 反代 | 第 5 阶段 |
| 只对外暴露 80、撤掉 3000/3306 外部映射 | 第 5 阶段 |
| 分服务 build / 滚动更新练习 | 第 6 阶段 |

> 第 3 阶段就是「写 compose、迁移、up -d、练 logs/ps/exec」四件事。心智越简单越好。

### 0.4 ⚠️ 再说一遍:`DB_HOST` 为什么改成 `mysql`?

第 2 阶段 MySQL 是 `docker run --name deploy-mysql` 起的,所以 server 里写 `DB_HOST=deploy-mysql`。

第 3 阶段 MySQL 在 compose 里是一个 **service**,服务名我们定为 `mysql`:

```yaml
services:
  mysql:          # ← 这个字符串就是 Docker 内部 DNS 名
    image: mysql:8.0
  server:
    environment:
      DB_HOST: mysql   # ← 必须和上面的服务名一致
```

**原理和第 2 阶段完全相同:写「对方在内部网络里的名字」,别写 `localhost`。** 只是名字从容器名 `deploy-mysql` 换成了 compose 服务名 `mysql`——这也是总计划第 2 阶段就预告的「应该写 mysql」。

## 二、写 `docker-compose.yml`(Win11 本地)

> compose 文件放在**项目根目录**(与 `frontend/`、`server/` 同级),和总计划第八章目录结构一致。Win11 写好后再传到 hp-linux。

### 2.1 新建 `docker-compose.yml`

在仓库根目录 `DeployLab/docker-compose.yml`:

```yaml
# Compose 项目名 → 默认网络名 deploy-lab_default、容器名前缀 deploy-lab-
name: deploy-lab

services:
  mysql:
    image: mysql:8.0
    restart: unless-stopped
    environment:
      MYSQL_ROOT_PASSWORD: "123456"
      MYSQL_DATABASE: deploy_lab
    ports:
      - "3306:3306"                    # ⚠️ 临时对外,第 5 阶段撤掉
    volumes:
      # ⭐ 复用第 1 阶段已有的 volume,数据不丢
      - deploy_mysql_data:/var/lib/mysql
      # volume 非空时 init 不会跑;留着是为了全新环境也能自动建表
      - ./server/db/init.sql:/docker-entrypoint-initdb.d/init.sql:ro

  server:
    build:
      context: ./server
      # hp-linux 区分大小写:若文件名是 DockerFile(大写 F),这里要写对
      dockerfile: DockerFile
    image: deploy-lab-server:v1
    restart: unless-stopped
    depends_on:
      - mysql
    ports:
      - "3000:3000"                    # ⚠️ 临时对外,方便 Win11 Vite;第 5 阶段撤掉
    environment:
      PORT: "3000"
      DB_HOST: mysql                    # ⭐ compose 服务名,不是 deploy-mysql
      DB_PORT: "3306"
      DB_USER: root
      DB_PASSWORD: "123456"
      DB_NAME: deploy_lab
      APP_VERSION: v1.0.0

volumes:
  deploy_mysql_data:
    external: true                      # 声明:用宿主机上已存在的 volume,compose 不新建
```

### 2.2 逐段理解(本阶段必须看懂)

```txt
name: deploy-lab
  → 项目名。影响默认网络名、容器名前缀。后面 docker compose ps 看到的 NAME 类似 deploy-lab-server-1

services.mysql / services.server
  → 两个服务。服务名 mysql / server 就是容器间 DNS 名

depends_on: [mysql]
  → server 在 mysql 容器 start 之后再 start(不保证 mysql 已 ready 接受连接;连不上时 docker compose restart server)

volumes.deploy_mysql_data.external: true
  → ⭐ 关键:绑定第 1 阶段创建的 deploy_mysql_data,而不是 compose 新建一个空 volume

build.context: ./server
  → compose build 时在 server/ 目录找 Dockerfile/DockerFile 打镜像

environment.DB_HOST: mysql
  → 和第 2 阶段的 deploy-mysql 同理,只是换成了 compose 服务名
```

### 2.3 关于密码要不要进 compose 文件?

本阶段为学习方便,密码直接写在 yml 里(和第 1、2 阶段 `-e` 注入一致)。**不要把这个文件提交到公开仓库**;若以后要进 git,可改用 `.env` + compose 的 `env_file:`(第 4 阶段以后可再收紧)。

### 2.4 (可选)给 MySQL 加 healthcheck,减少 server 首启连库失败

若 `docker compose up -d` 后 server 日志里偶发 `ECONNREFUSED`,多半是 mysql 还在初始化。可在 `mysql` 服务下加:

```yaml
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "127.0.0.1", "-uroot", "-p123456"]
      interval: 5s
      timeout: 5s
      retries: 10
      start_period: 30s
```

并把 `server` 的 `depends_on` 改成:

```yaml
    depends_on:
      mysql:
        condition: service_healthy
```

> 不加也能完成本阶段;加了更接近生产写法,值得练。

## 四、迁移:停掉第 2 阶段的手工容器(hp-linux)

> ⭐ 顺序很重要:**先停 server,再停 mysql;只删容器,不删 volume。**

### 4.1 停并删旧容器

```bash
ssh hp-linux-ts
cd ~/code/person/deploy-lab

# 1) 先停 server(它依赖 mysql)
docker stop deploy-server
docker rm deploy-server

# 2) 再停 mysql
docker stop deploy-mysql
docker rm deploy-mysql

# 3) 确认 volume 还在(必须有)
docker volume ls | grep deploy_mysql_data
```

成功标志:`docker ps -a` 里**没有** `deploy-server`、`deploy-mysql`;`deploy_mysql_data` volume **仍在**。

> ⚠️ **绝对不要**在这一步执行 `docker volume rm deploy_mysql_data`,否则第 1 阶段起的所有任务数据都会丢。

### 4.2 (可选)清理第 2 阶段的手工网络

```bash
docker network rm deploy-lab-net    # 若提示 in use,说明还有容器挂着,先 docker ps -a 排查
```

Compose 会自己建 `deploy-lab_default`,旧的 `deploy-lab-net` 不再需要。

### 4.3 确认 3000 / 3306 端口已释放

```bash
ss -ltnp | grep -E '3000|3306'    # 应无输出(或只有即将被 compose 占用的)
```

## 七、Compose 核心知识点(搞懂就不怕改 yaml)

### 7.1 `docker compose` 和 `docker run` 的对应关系

| 第 2 阶段 `docker run` | 第 3 阶段 compose |
|------------------------|-------------------|
| `--name deploy-server` | `services.server`(服务名 server) |
| `--network deploy-lab-net` | compose 自动把所有 service 放进同一默认网络 |
| `-e DB_HOST=deploy-mysql` | `environment.DB_HOST: mysql` |
| `-p 3000:3000` | `ports: ["3000:3000"]` |
| `--restart unless-stopped` | `restart: unless-stopped` |
| `deploy-lab-server:v1` | `image:` + 可选 `build:` |
| `-v deploy_mysql_data:...` | `volumes:` 段 + `external: true` |

### 7.2 为什么 compose 里 MySQL 服务名必须叫 `mysql`?

总计划最终形态、第 4 阶段 Redis、第 5 阶段 Nginx 反代,都约定:

```txt
server → mysql:3306
server → redis:6379
nginx  → server:3000
```

服务名就是**稳定的内部 hostname**。你可以改成别的(比如 `db`),但 `DB_HOST` 必须跟着改;本项目统一用 `mysql`,和总计划对齐。

### 7.3 `external: true`  volume 在迁移里扮演什么角色?

```txt
第 1 阶段 docker run -v deploy_mysql_data:...  → 数据写进这个 volume
第 3 阶段停掉 deploy-mysql 容器              → 容器没了,volume 还在
compose up 新 mysql 容器                      → 挂载同一个 deploy_mysql_data
                                              → tasks 表、历史数据全部保留
```

若忘了写 `external: true`,compose 会新建一个叫 `deploy-lab_deploy_mysql_data` 的空 volume,MySQL 会是**空库**,health 可能 ok 但 tasks 全没——这是本阶段最常见的数据「丢失」错觉。

### 7.4 `depends_on` 不等于「数据库已就绪」

`depends_on` 只保证**启动顺序**,不保证 MySQL 已完成 init、已开始监听 3306。生产环境常配合 `healthcheck` + `condition: service_healthy`(见 2.4)。本阶段若首启 `db:down`,等 10~20 秒后:

```bash
docker compose restart server
curl -s http://localhost:3000/api/health
```

## 九、常见坑速查表

| 现象 | 大概率原因 | 怎么查 / 怎么修 |
|------|-----------|----------------|
| `up` 报 `volume deploy_mysql_data not found` | hp-linux 上从未建过该 volume(或名字拼错) | `docker volume ls`;若真没有,说明没走过第 1 阶段,需先按 0004 起 MySQL |
| tasks 全空、像新库 | compose 新建了 volume,没绑旧的 | compose 里 `external: true` + 正确 volume 名;`docker volume ls` 对比 |
| `db:down`、`ENOTFOUND mysql` | `DB_HOST` 写错,或服务名与 yml 不一致 | 核对 `services.mysql` 与 `DB_HOST`;`docker compose exec server nslookup mysql` |
| `db:down`、`ECONNREFUSED 127.0.0.1` | `DB_HOST=localhost` | 改成 `mysql` 后 `docker compose up -d server` |
| `build` 报找不到 Dockerfile | Linux 大小写:文件是 `DockerFile` | compose 里 `dockerfile: DockerFile` 或重命名为 `Dockerfile` |
| `3306` / `3000` 端口被占用 | 旧容器没删干净 | `docker ps -a`;`ss -ltnp \| grep 3306` |
| 首启 server `db:down`,过一会又好 | mysql 还在 init | 等 20s 或 `docker compose restart server`;或加 2.4 healthcheck |
| Win11 curl 不通 | 3000 未映射 / Tailscale 断 | `docker compose ps` 看 PORTS;`tailscale status` |
| `docker compose exec mysql` 进不去 | mysql 服务没起来 | `docker compose ps`;`docker compose logs mysql` |

### 11.2 给后面阶段埋的伏笔(心里有数即可)

