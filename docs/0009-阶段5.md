# DeployLab 阶段五：加入前端和 Nginx · 执行方案

> 对应 `0001-总计划.md` 第九章「第 5 阶段：加入前端和 Nginx」。
> 一句话目标：**在 Compose 里再加一个 `frontend` 服务（本质 = Nginx + Vue 的 dist），让浏览器访问「服务器 IP:80」直接看到页面，`/api` 由 Nginx 自动反代到 `server:3000`；同时把 `server:3000`、`mysql:3306` 的外部端口映射撤掉，只留 80 对外——完成「安全收口」。**
> 这一阶段没有迁移、没有数据风险：前端是全新的无状态容器，MySQL volume 纹丝不动。
> 最重要的心智：**Nginx 反代 `/api` 时 `proxy_pass` 后面写的是 Compose 服务名 `server`，不是 `localhost`，也不是阶段 2 的旧容器名 `deploy-server`**——和 `DB_HOST=mysql`、`REDIS_HOST=redis` 是同一堂「写内部 DNS 名」的课。

---

## 〇、阶段定位与边界

### 0.1 这一阶段的部署形态

阶段 4 是 `mysql + redis + server` 三个服务，server 的 3000 还映射到宿主机给 Win11 跨机调试。阶段 5 在同一份 `docker-compose.yml` 里**多加一个 `frontend` 服务，并把 3000/3306 的对外映射撤掉**，最终只有 80 端口面向外网：

```txt
Win11 开发机                                hp-linux server (Ubuntu 24.04)
┌────────────────────────┐                ┌─────────────────────────────────────────────────┐
│ (可选) Vue dev :5173    │                │  Compose 项目:deploy-lab                         │
│  └ Vite proxy /api      │ ─Tailscale──▶ │  默认网络:deploy-lab_default(阶段3 已建)         │
│     → 100.98.102.107    │   :80(不再是   │  ┌───────────┐                                   │
│        :80              │    :3000)      │  │ frontend │  Nginx 80                        │
└────────────────────────┘                │  │ Vue dist │──── /api 反代 ──┐                  │
                                          │  │  :80 ⭐  │                  ▼                  │
                                          │  └───────────┘           ┌──────────┐            │
                                          │   -p 80:80(唯一对外)     │  server  │            │
                                          │                          │ Express  │            │
                                          │                          │  :3000   │            │
                                          │                          └────┬─────┘            │
                                          │                  ┌──────────┴──────────┐         │
                                          │                  ▼                      ▼         │
                                          │            ┌──────────┐           ┌──────────┐    │
                                          │            │  mysql   │           │  redis   │    │
                                          │            │  :3306   │           │  :6379   │    │
                                          │            └──────────┘           └──────────┘    │
                                          │   3000 / 3306 / 6379 都不再映射到宿主机          │
                                          └─────────────────────────────────────────────────┘
```

请求流向（和阶段 4 比，**到 server 之后的链路完全不变**，变的只是「前端怎么到 server」——从「浏览器直连 3000」变成「浏览器到 80，Nginx 再转 3000」）：

```txt
浏览器 ─▶ 100.98.102.107:80 ─▶ frontend(Nginx) ┬─ 静态请求(/ , /index.html, /xxx) → Vue dist（SPA fallback）
                                                └─ /api/* ─▶ server:3000 ┬─ mysql:3306【读写任务】
                                                                         └─ redis:6379【计数 + ping】
```

### 0.2 与第 4 阶段的关键差异

| 维度 | 第 4 阶段 | 第 5 阶段（本方案） |
|------|-----------|--------------------|
| Compose 服务数量 | 3（mysql、redis、server） | **4（mysql、redis、server、frontend）** |
| 前端怎么跑 | Win11 本地 `npm run dev`，:5173 | **容器化，Nginx 托管 dist，:80** |
| `/api` 谁代理 | Vite dev server 的 proxy（仅本地开发用） | **Nginx 的 `location /api/` 反代（生产入口）** |
| 对外端口 | 3000、3306 映射到宿主机 | **只映射 80；3000/3306/6379 全部收回** |
| 浏览器访问地址 | Win11 上 `localhost:5173`（或直连 `IP:3000` 调 API） | **任意机器访问 `服务器IP:80` 直接出页面** |
| server / mysql 是否暴露公网 | 是（临时） | **否（只内网可达）** |
| 前端镜像 | 没有 | **新增 `deploy-lab-frontend:v1`（多阶段构建）** |

本阶段一上线，总计划第十章验收标准里的 **#2「浏览器访问服务器 IP 能看到前端」、#6「只有 80 端口对外开放」、#7「server/mysql 不直接暴露公网」** 三条一并达成。

### 0.3 这一阶段「不做什么」（防止跑偏）

| 不做的事 | 留到哪个阶段 |
|---------|-------------|
| 改前端/后端代码做「发布更新」练习（改标题、改接口、加字段） | 第 6 阶段 |
| HTTPS / 443 / 证书 | 本项目不做（学习环境 HTTP 够用，见 7.7） |
| Nginx 负载均衡、多副本 | 本项目不做（单副本够练） |
| 前端路由改用 vue-router 多页 | 本项目就一个单文件 App.vue，SPA fallback 够练 |
| CDN、gzip、缓存头精细化 | 本项目不做（默认 nginx 配置够用） |

> 第 5 阶段就是「加 frontend 服务、Nginx 托管 + 反代、撤 3000/3306」三件事。心智越简单越好。

### 0.4 ⚠️ 前端容器的本质 = Nginx + dist

这是本阶段最核心的一句话，总计划也专门点了它。前端「容器化」不是把 `npm run dev` 装进容器，而是：

```txt
1. 构建阶段（builder）：node 镜像里 npm ci + npm run build → 产出 dist/ 静态文件
2. 运行阶段（runtime）：把 dist/ 塞进 nginx 镜像，用 nginx 托管它
```

所以**最终运行的容器里没有 Node、没有 Vite、没有源码，只有 Nginx + 一堆静态文件（html/js/css）**。和「把 dist 丢到任意静态服务器」是等价的——容器只是把这一步标准化了。

好处：构建产物和运行环境解耦，镜像小（nginx:alpine ~50MB），启动秒级，和后端用同一套 Compose 编排。

> 仓库里的 `frontend/Dockerfile` 已经按这个多阶段写好了（见 1.4 摸底），本阶段主要是「读懂它 + 接进 compose」。

### 0.5 ⚠️ `proxy_pass` 写服务名 `server`，不是 `localhost` / `deploy-server`

Nginx 在容器里要把 `/api` 转给后端，写的是 **Compose 内部 DNS 名**：

```nginx
location /api/ {
    proxy_pass http://server:3000;     # ← server 是 compose 服务名，不是 localhost
}
```

和阶段 2/3/4 的 `DB_HOST=mysql`、`REDIS_HOST=redis` 完全同理：

```txt
写 localhost    → Nginx 容器自己没有后端，ECONNREFUSED 127.0.0.1:3000，页面 /api 报 502
写 deploy-server → 阶段 2 的旧容器名，compose 网络里没有这个名字，解析失败，502
写 server       → ✅ compose 网络 DNS 解析到 server 容器，反代成功
```

> ⚠️ **仓库里现成的 `frontend/nginx.conf` 写的就是 `deploy-server`（阶段 2 残留）**——这是一个已知 bug，第二章会先把它修掉，否则 up 之后 `/api` 全部 502。

### 0.6 安全收口：为什么撤掉 3000 / 3306？

阶段 3/4 临时把 server 的 3000、mysql 的 3306 映射到宿主机，是**为了 Win11 跨机调试方便**。到阶段 5，前端成了统一入口，这两个映射就没必要了，而且留着是**安全隐患**：

```txt
留着 3006/3306 → 任何能 ping 通服务器 IP 的人都能直连数据库 / 直连后端
撤掉           → server、mysql 只存在于 deploy-lab_default 内网
                → 外部唯一入口是 80（Nginx），Nginx 只放行「静态文件」和「/api」
                → 想查数据库？只能 ssh 上服务器 docker compose exec（见 8.3）
```

这正是总计划验收 #6/#7 要求的形态，也是阶段 4「Redis 6379 从一开始就不外露」的延伸——**现在把另外两个也收进来，对外只剩一个口子**。

### 0.7 跨机访问方式变了（重要，影响你以前的调试习惯）

| 操作 | 阶段 4（3000 还开着） | 阶段 5（3000 已关） |
|------|----------------------|---------------------|
| Win11 浏览器看页面 | `localhost:5173`（本地 dev） | **`http://100.98.102.107`（直接访问 80）** |
| Win11 curl 健康 | `curl 100.98.102.107:3000/api/health` | **`curl 100.98.102.107/api/health`（80）** |
| hp-linux 本机 curl | `curl localhost:3000/api/health` | **`curl localhost/api/health`**，或 `docker compose exec server wget -qO- localhost:3000/api/health` |
| Win11 直连 MySQL（ navicat 等） | `IP:3306` 能连 | **连不上了**，只能 `docker compose exec mysql mysql ...` |

> 一句话：**以前直连后端端口，现在一律走 80 这个大门。** 后台数据/服务只能 ssh 进服务器后用 `docker compose exec` 碰——这正是生产环境的常态。

逐段理解：

```txt
FROM node:20-alpine AS builder
  → 第一阶段叫 builder，用 node 镜像装依赖 + 构建

COPY package*.json + npm ci（先于 COPY . .）
  → 老套路：依赖描述文件单独一层，命中缓存，改源码 rebuild 时不重装依赖

RUN npm run build
  → vue-tsc -b && vite build，产出 /app/dist

FROM nginx:alpine
  → 第二阶段换 base 镜像为 nginx，前面的 node/源码都不会进最终镜像

COPY --from=builder /app/dist /usr/share/nginx/html
  → 只把 builder 阶段的 dist 拷过来 → 最终镜像里没有 node、没有源码，只有静态文件

EXPOSE 80 + nginx -g "daemon off;"
  → nginx 前台运行（容器主进程不能是守护进程，否则容器会立即退出）
```

> 这就是 0.4 说的「Nginx + dist」的落地。本阶段不需要改 Dockerfile，读懂即可。

**`frontend/.dockerignore`（已就位，很关键）：**

```text

dist
.git
.env
.env.local
*.log
```

理解：

```txt
node_modules  → 宿主机的 node_modules 会被 COPY . . 拷进去覆盖容器里的，必须忽略
```

## 二、修 `frontend/nginx.conf`（关键：修 proxy_pass 的 bug）

> 这是本阶段**唯一必须改的前端文件**。现成的配置把 `/api` 转给了阶段 2 的旧容器名 `deploy-server`，在 Compose 网络里解析不到，必须改成服务名 `server`。

### 2.1 改 `proxy_pass`：`deploy-server` → `server`

`frontend/nginx.conf` 完整内容（改后）：

```nginx
server {
    listen 80;
    server_name _;

    root /usr/share/nginx/html;
    index index.html;

    # SPA 路由回退：所有未匹配静态文件的路径都交给 index.html
    location / {
        try_files $uri $uri/ /index.html;
    }

    # API 请求反向代理到后端服务
    location /api/ {
        proxy_pass http://server:3000;          # ⭐ 改这里：deploy-server → server
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

### 2.2 三段配置逐行理解

```txt
listen 80;  server_name _;
  → 监听 80，接收所有域名（_ 是通配）。容器里 nginx 就这一个 server 块

root /usr/share/nginx/html;  index index.html;
  → 静态文件根目录，正是 Dockerfile 里 COPY dist 进去的那个目录

location / { try_files $uri $uri/ /index.html; }
  → SPA fallback（见 7.3）：先找真实文件，找不到就回 index.html，交给 Vue 路由

location /api/ { proxy_pass http://server:3000; ... }
  → 凡是 /api/ 开头的请求，转给 server:3000（compose 服务名）
  → 下面 4 行 proxy_set_header：把真实客户端信息透传给后端，否则 server 看到的 IP 永远是 nginx
```

### 2.3 ⚠️ `proxy_pass` 末尾**不要加斜杠**

这是一个非常容易踩、又非常隐蔽的坑：

```nginx
# ✅ 正确：不带 URI（末尾无斜杠）→ 原样转发 /api/tasks
proxy_pass http://server:3000;

# ❌ 错误：带斜杠 → 会把 /api/ 前缀「吃掉」，转发成 /tasks，后端 404
proxy_pass http://server:3000/;
```

Nginx 的规则：**`proxy_pass` 后面只要带了 URI（哪怕只是一个 `/`），就会用它替换掉 `location` 匹配到的前缀**。

```txt
请求：GET /api/tasks
location /api/  +  proxy_pass http://server:3000    → server 收到 /api/tasks  ✅（server 路由挂在 /api）
location /api/  +  proxy_pass http://server:3000/   → server 收到 /tasks      ❌（404）
```

> 后端 `app.use('/api/tasks', ...)` 是挂在 `/api` 下的，所以 nginx 必须**原样保留 `/api` 前缀**——也就是 `proxy_pass` 不带末尾斜杠。记不住没关系，记住一句：**`proxy_pass http://服务名:端口;` 后面直接分号，别画蛇添足加 `/`。**

## 三、（建议）调整 `vite.config.ts` 的 proxy 目标

> 这一步**只影响 Win11 本地 `npm run dev` 调试**，不影响容器。但因为阶段 5 要关闭 3000 端口，原来 proxy 指向的 `:3000` 会失效，顺手改掉。

### 3.1 把 target 从 `:3000` 改到 `:80`

`frontend/vite.config.ts`：

```ts
import { defineConfig } from 'vite'
import vue from '@vitejs/plugin-vue'

export default defineConfig({
  plugins: [vue()],
  server: {
    port: 5173,
    proxy: {
      '/api': {
        target: 'http://100.98.102.107',   // ⭐ 从 :3000 改成 :80（端口默认 80 可省略）
        changeOrigin: true,
      },
    },
  }
})
```

理解：

```txt
阶段 4：target = http://100.98.102.107:3000  → 直连 server 容器
阶段 5：target = http://100.98.102.107       → 改走 nginx:80，nginx 再转 server:3000

本地 dev 请求链路变成：
  浏览器 localhost:5173 ─(Vite proxy)─▶ 100.98.102.107:80 ─(nginx /api)─▶ server:3000
```

> 多了一跳，但本地 dev 本来就是辅助手段。**主入口已经是「浏览器直接访问 100.98.102.107:80」**，本地 dev 只是改前端样式时图个热更新。

> 如果你这阶段完全不打算在 Win11 跑 `npm run dev`，这一步可以跳过——但建议改了，免得下次本地启动时 proxy 报 `ECONNREFUSED :3000` 一脸懵。

## 四、改 `docker-compose.yml`（核心）

> 改动 3 处：① 新增 `frontend` 服务；② 删掉 server 的 `ports: 3000`；③ 删掉 mysql 的 `ports: 3306`。Redis、server 的环境变量、MySQL volume **一个字都不改**。

### 4.1 完整的 `docker-compose.yml`（阶段 5 终态）

```yaml
name: deploy-lab

services:
  mysql:
    image: mysql:8.0
    restart: unless-stopped
    environment:
      MYSQL_ROOT_PASSWORD: "123456"
      MYSQL_DATABASE: deploy_lab
    # ⭐ 阶段5：撤掉 ports: "3306:3306"，mysql 只在内网暴露
    volumes:
      - deploy_mysql_data:/var/lib/mysql
      - ./server/db/init.sql:/docker-entrypoint-initdb.d/init.sql:ro
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "127.0.0.1", "-uroot", "-p123456"]
      interval: 5s
      timeout: 5s
      retries: 10
      start_period: 30s

  redis:
    image: redis:7-alpine
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 5s
      retries: 10
      start_period: 30s

  server:
    build:
      context: ./server
      dockerfile: DockerFile
    image: deploy-lab-server:v2
    restart: unless-stopped
    depends_on:
      mysql:
        condition: service_healthy
      redis:
        condition: service_healthy
    # ⭐ 阶段5：撤掉 ports: "3000:3000"，server 只在内网暴露（由 nginx 反代）
    environment:
      PORT: "3000"
      DB_HOST: mysql
      DB_PORT: "3306"
      DB_USER: root
      DB_PASSWORD: "123456"
      DB_NAME: deploy_lab
      REDIS_HOST: redis
      REDIS_PORT: "6379"
      APP_VERSION: "1.0.0"

  # ⭐ 新增：前端服务（本质 = Nginx + Vue dist）
  frontend:
    build:
      context: ./frontend
      dockerfile: Dockerfile
    image: deploy-lab-frontend:v1            # 首个前端镜像版本
    restart: unless-stopped
    depends_on:
      server:
        condition: service_started           # 见 4.3：server 无 healthcheck，用 started
    ports:
      - "80:80"                              # ⭐ 唯一对外的端口

volumes:
  deploy_mysql_data:
    external: true
```

### 4.2 逐段理解（只看新增 / 变化部分）

```txt
services.frontend
  → 新的 service。服务名 frontend 即内部 DNS 名（nginx 用 server 名反代，自己叫什么不影响别人）

build.context: ./frontend + dockerfile: Dockerfile
  → 用 frontend/ 下的多阶段 Dockerfile 构建（见 1.4）

image: deploy-lab-frontend:v1
  → 给镜像打 tag，方便第 6 阶段演练「改前端 → build → 换版本号 → 回滚」

depends_on.server.condition: service_started
  → frontend 等 server 容器「启动」即可（不必等 healthy，原因见 4.3）

ports: "80:80"
  → 把容器的 80 映射到宿主机 80，这是本项目唯一面向公网的端口
```

server / mysql 的变化：

```txt
删掉 server 的 ports: "3000:3000"
  → server 仍监听容器内 3000，但宿主机/外网连不到，只有同网络的 nginx 能连 server:3000

删掉 mysql 的 ports: "3306:3306"
  → 同理，mysql 只能被 server 在内网访问；排错用 docker compose exec（见 8.3）
```

### 4.3 ⚠️ 为什么 frontend 的 `depends_on` 用 `service_started` 而不是 `service_healthy`？

`condition: service_healthy` 要求**被依赖的服务定义了 `healthcheck`**。但当前 compose 里 `server` **没有 healthcheck**（只有 mysql、redis 有）。如果写 `service_healthy`，compose 会报错：

```txt
Dependency "server" does not have a healthcheck configured
```

所以这里用 `service_started`——server 容器进程一拉起，frontend 就启动。即使 server 还没完全 ready，nginx 也能起来；这时若有 `/api` 请求进来，nginx 会返回 502，等几百毫秒 server 就绪后自动恢复。学习环境完全可接受。

> （可选）若想让 frontend 也等 server 真正 ready，可以给 server 加一个 healthcheck（`wget --spider localhost:3000/api/health`），然后把 condition 改回 `service_healthy`。本阶段不做，保持简单。

### 4.4 关于服务命名的一个小说明

总计划第四章的结构图里把它叫 `frontend-nginx 容器`（强调它是个 nginx）。本方案把 **compose 服务名定为 `frontend`**——理由：

- 服务名是给「同网络其它服务」和「docker compose 命令」用的；没有任何配置引用 frontend 的名字（nginx 反代的是 `server`，不是 frontend）；
- 把实现细节（nginx）写进服务名是反模式，万一以后换成 caddy，名字就得改；
- 服务名短一点，敲 `docker compose logs frontend` 更顺。

> 容器名会是 `deploy-lab-frontend-1`（compose 自动加项目名前缀），和「frontend-nginx」这个叫法不冲突——它本质上就是个跑 nginx 的前端容器。

## 八、核心知识点（搞懂就懂「前端部署」这一层）

### 8.1 多阶段构建（multi-stage build）省了什么？

```txt
单阶段（错误示范）：FROM node:20 → 装依赖 → build → 运行也用 node
  → 最终镜像里有 node_modules(几百 MB) + 源码 + 构建工具，体积大、攻击面大
  → 而且运行时根本用不到 node，只需要静态文件

多阶段（本项目的做法）：
  阶段1 builder(node)：装依赖 + build，产出 dist
  阶段2 runtime(nginx)：只 COPY dist 过来
  → 最终镜像 ≈ nginx:alpine(50MB) + dist(几 MB)，没有 node、没有源码
```

> 一句话：**构建用的工具（node/vite）和运行用的环境（nginx）分离。** 这也是后端镜像为啥小（server 运行时也需要 node，所以它没分阶段，但用了 `--omit=dev` 去掉 nodemon 等开发依赖）。

### 8.2 Nginx 在这里干了三件事

```txt
① 托管静态文件     root + index → 把 Vue 的 dist 当网站提供出去
② SPA fallback    try_files    → 刷新子路由不 404，交给前端路由
③ 反向代理 /api    proxy_pass   → 把接口请求转给 server，对外只露 80
```

**这三件事合起来，就是「前端容器 = 统一入口」的全部职责**。总计划第三章列的「Nginx 练：托管前端 / 反向代理 /api / SPA fallback / 统一入口 / 只暴露 80」全部覆盖。

### 8.3 SPA fallback 为什么必须有？

Vue（含 vue-router）是单页应用：无论 URL 是 `/`、`/about` 还是 `/tasks/3`，**实际都只有一份 `index.html`**，路由是 JS 在前端解析的。

```txt
用户在浏览器里点链接  → JS 拦截、前端路由切换 → 不发请求（没事）
用户在地址栏刷新      → 浏览器向服务器真的请求 /about
                       → nginx 找不到 /about 这个文件
                       → 没 try_files：返回 404 ❌
                       → 有 try_files ... /index.html：返回 index.html，前端路由接管 ✅
```

> 本项目只有一个页面看不出区别，但 `try_files` 是 SPA 部署的「标配」，先养成习惯。

### 8.4 反向代理（reverse proxy）是什么？

```txt
正向代理：客户端知道要访问谁，代理替你去（如科学上网）—— 代理替「客户端」工作
反向代理：客户端不知道真正的后端是谁，以为 nginx 就是服务器 —— 代理替「服务端」工作
```

本项目的 nginx 就是反向代理：浏览器以为 `100.98.102.107/api/tasks` 是 nginx 在响应，**实际 nginx 把请求偷偷转给了 `server:3000`**。好处：

- 对外只有 80 一个口，server/mysql/redis 全藏在内网（安全）；
- 以后换后端（比如 server 改成 Go 重写），只要 nginx 的 `proxy_pass` 不变，前端无感（解耦）；
- 可以在 nginx 这层加日志、限流、缓存（扩展性）。

### 8.5 同源之后，CORS 还需要吗？

阶段 4 前后端跨域（Vite :5173 → server :3000），所以后端有 `app.use(cors())`。阶段 5 后，**浏览器看到的页面和 `/api` 都来自同一个 `http://100.98.102.107:80`**——同源，不再有 CORS 问题。

那 `cors()` 能不能删？

- 删了：容器化访问（80）完全正常；但 **Win11 本地 `npm run dev`（:5173 → :80）会跨域**，本地调试就废了；
- 留着：无害（同源请求浏览器根本不发预检），本地 dev 也兼容。

> **建议留着**——学习项目优先保证「本地 dev + 容器 prod」都能跑。生产环境真要收紧，可以在 nginx 层或后端按需配置，本项目不展开。

### 8.6 为什么前端用相对路径 `/api`，不用绝对地址？

看 `App.vue`：

```ts
const API = (p) => fetch('/api' + p, ...).then(...)
```

`/api` 是**相对路径**，浏览器会自动拼成「当前页面所在 origin + /api」：

```txt
本地 dev（origin localhost:5173） → 请求 localhost:5173/api → Vite proxy 转 80 → nginx → server
容器 prod（origin 100.98.102.107:80） → 请求 100.98.102.107/api → nginx → server
```

**同一份前端代码，两种环境都通，不用改、不用配环境变量。** 这就是为什么阶段 5 前端代码一个字都不用动——因为请求地址写的是相对的 `/api`，谁托管它，`/api` 就归谁代理。

> 如果当初写成 `fetch('http://100.98.102.107:3000/api...')`（硬编码绝对地址），换环境就得改代码——这是反模式。相对路径是 SPA 对接反向代理的最佳实践。

```txt
后续可选项（不在计划内）：
  - HTTPS：给 nginx 加 443 + 证书，把 80 重定向到 443
  - 前端环境变量：VITE_ 前缀的变量打进 dist（阶段5 没用到，业务简单）
  - 给 server 加 healthcheck，让 frontend 能 condition: service_healthy
```

### 12.3 到这里，总计划验收十条达成情况

```txt
1. docker compose up -d 能启动            ✅（阶段3 起，阶段5 扩到 4 服务）
2. 浏览器访问 IP 能看到前端                ✅（阶段5 达成）
3. 前端能调用 /api/tasks                   ✅（阶段5 经 nginx 反代达成）
4. 后端能读写 MySQL                        ✅（阶段1 起）
5. MySQL 数据重启容器不丢                   ✅（阶段3 volume）
6. 只有 80 端口对外开放                     ✅（阶段5 收口达成）
7. server/mysql 不直接暴露公网              ✅（阶段5 收口达成）
8. docker compose logs 能看到日志           ✅（阶段3 起）
9. 改前端后能重新 build 发布                ⏳（阶段5 已预演，阶段6 正式练）

```

## 十三、和第 4 阶段命令对照速查

阶段 5 没有引入新命令体系，日常操作还是同一套 Compose，只是「多了个 frontend、少了对外的 3000/3306」：

| 操作 | 第 4 阶段 | 第 5 阶段（多/变） |
|------|----------|-------------------|
| 看服务状态 | `docker compose ps`（3 行） | **4 行（多 frontend）；PORTS 列只剩 80** |
| 看前端日志 | —（前端在 Win11 本地） | **`docker compose logs frontend`**（nginx access log） |
| 进 nginx 容器 | — | **`docker compose exec frontend sh`**（看 `/etc/nginx/conf.d/default.conf`、`/usr/share/nginx/html`） |
| 看后端健康 | `curl localhost:3000/api/health` | **`curl localhost/api/health`**（80） |
| 查 MySQL | `mysql -h localhost -P 3306`（宿主） | **只能 `docker compose exec mysql mysql ...`**（3306 收回） |
| 重建前端镜像 | — | **`docker compose build frontend`**（首次慢，后续命中缓存快） |
| 改 nginx 配置生效 | — | 改 `nginx.conf` → `docker compose build frontend && docker compose up -d frontend` |
| 停启单服务 | `docker compose stop redis` | 同左（可对 server 做 502 实验、对 frontend 做页面不可达实验） |
| 数据回归检查 | `... SELECT COUNT(*) FROM tasks` | 同左（本阶段不动 MySQL） |

> 一句话：**阶段 5 的世界多了个 Nginx 当大门，也顺手把后门全关了。** 对外只剩一个 80，里面四个容器各司其职——这正是「生产级」部署的雏形，也是 Compose 编排的最终回报。
