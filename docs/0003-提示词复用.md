# 提示词复用

> 这一页是「可以直接复制粘贴给 AI」的提示词模板。
> 用法:登上 hp-linux server 后,把对应方向的提示词复制给 AI,它会告诉你该跑什么命令、怎么解读结果。
> 所有提示词都围绕 `0001-总计划.md` 的部署目标设计,确保你在部署 DeployLab 之前/之中/之后,对这台服务器有完整、必要的了解。

---

## hp-linux server常用提示词

服务器基本信息(已确认):

```txt
系统:Ubuntu 24.04.4 LTS (内核 6.17)
登录:ssh hp-linux-ts  (密码见 0002-信息资料.md)
Tailscale IP:100.98.102.107
WiFi IP:192.168.66.107
当前已运行的 Docker 容器:Web(8020) / MySQL(3306) / Redis(6379)
已接入服务:docker、containerd、ssh、tailscaled
```

下面按「部署前 → 部署中 → 部署后」分 10 个方向,每个方向给一个可复用提示词,并标注它对应总计划的哪个阶段。

---

### 方向 1:环境盘点 —— 这台机器现在到底在跑什么

> 对应总计划:第 1 阶段开始前,先摸清家底。

```txt
我刚 SSH 登录一台 Ubuntu 24.04 服务器,准备在上面部署一个
Vue+Express+MySQL+Redis+Nginx 的项目。
请给我一组命令,帮我完整盘点这台服务器当前的状态,包括:
1. 系统版本、内核、CPU、内存、磁盘可用空间
2. 当前正在运行的 systemd 服务
3. 当前 Docker 正在运行的容器和占用的端口
4. 当前监听网络的端口清单
请把命令分类列出,并说明每条命令输出的关键字段我该看什么。
```

---

### 方向 2:Docker / Compose 环境就绪度

> 对应总计划:第 2~5 阶段全部依赖 Docker。

```txt
我要在这台服务器上用 docker compose 部署多容器项目。请给我命令检查:
1. docker 是否安装、版本号、是否正常运行(systemctl status docker)
2. docker compose(v2 插件)是否可用、版本号
3. 当前用户是否在 docker 组(能否免 sudo 运行 docker)
4. containerd 状态
5. Docker 的存储驱动和根目录、磁盘占用
如果有缺失或配置不当,请告诉我怎么修。
```

---

### 方向 3:端口规划与冲突排查 ⚠️ 重点

> 对应总计划:第 5 阶段只对外暴露 80(验收标准第 6 条)。
> 注意:这台机器 3306 / 6379 / 8020 已被现有容器占用,DeployLab 必须避开它们。

```txt
我准备部署一个项目,计划使用的端口:
- 80   (Nginx 对外入口)
- 3000 (后端 Express,仅容器内)
- 3306 (MySQL)
- 6379 (Redis)
但已知这台机器的 3306、6379、8020 已被别的 Docker 容器占用。
请给我命令:
1. 检查 80、3000 端口是否空闲
2. 列出所有被占用的端口及对应进程/容器
3. 如果 3306/6379 冲突,告诉我有哪些解决办法(改端口 / 复用现有 / 隔离网络)
4. 帮我规划一套不冲突的端口方案,并给出对应的 docker-compose.yml ports 写法
```

---

### 方向 4:目录与数据存储规划

> 对应总计划:第 3 阶段 volume、验收标准第 5 条「重启容器数据不丢」。

```txt
我要在服务器上部署项目,代码放哪、MySQL 数据 volume 放哪需要规划。请给我命令和建议:
1. 查看磁盘分区和各挂载点剩余空间(df -h)
2. 看一下家目录结构,建议项目放在哪个路径
3. Docker 默认 volume 存在哪、当前已有哪些 volume
4. 给我一套目录规划建议(项目代码、docker volume、备份目录分别放哪)
5. 如果以后数据要迁移/备份,数据实际存在文件系统的哪个路径
```

---

### 方向 5:用户与权限

> 对应总计划:全程要用 ssh 登录执行命令。

```txt
我通过普通用户 vichel + sudo 登录这台服务器。请帮我确认:
1. 当前用户是谁、有哪些 sudo 权限(sudo -l)
2. 是否属于 docker 组
3. 文件/目录默认权限
4. 部署项目时,如何避免所有文件都是 root
   (常见坑:docker 容器生成的文件属主变 root,普通用户改不了)
```

---

### 方向 6:现有数据库/缓存现状

> 对应总计划:第 3、4 阶段要用 MySQL 和 Redis。已有容器需要决定「复用还是另起」。

```txt
这台机器上已经有 Docker 跑的 MySQL(3306)和 Redis(6379)。
我新项目也要用 MySQL 和 Redis。请帮我判断和决策:
1. 怎么查现有 MySQL 容器是什么镜像、什么版本、root 密码在哪、有哪些库
2. 怎么查现有 Redis 容器是否设了密码
3. 复用现有数据库 vs 新起独立容器,各自的优缺点和风险
4. 如果新起容器但要换端口,compose 该怎么写
```

---

### 方向 7:网络访问与防火墙

> 对应总计划:验收标准第 6、7 条「只暴露 80,server/mysql 不暴露公网」。
> 已知当前 3306/6379 监听 0.0.0.0,有风险。

```txt
这台服务器接入了 Tailscale(100.98.102.107),也在 WiFi 网(192.168.66.107)。
我希望部署后:外部只能通过 80 端口访问前端,数据库和 Redis 不对局域网暴露。
请给我:
1. 查看/配置防火墙(ufw)状态的命令
2. 一套 ufw 规则建议(放行 SSH、放行 Tailscale 网卡、只放行 80)
3. 如何让 docker 容器端口只绑 127.0.0.1,不监听 0.0.0.0
4. 如何通过 Tailscale 从我家里 win11 安全访问这台机器的管理端口
注意:给我命令时务必保证不会把我自己的 SSH 锁在外面。
```

---

### 方向 8:故障排查与日志定位

> 对应总计划:第 7 阶段故障排查练习。

```txt
部署过程中容器可能起不来。请给我一套通用的排查命令模板,覆盖:
1. 查看某服务/容器状态(docker ps / docker compose ps)
2. 看容器日志(docker logs、docker compose logs、follow + 指定时间)
3. 进容器内部排查(docker exec、检查网络连通性、能不能 ping 通 mysql/redis)
4. 端口是否真的在监听、防火墙是否拦了
5. 常见报错(DB_HOST 写错、密码错、proxy_pass 写错、端口被占)分别对应看哪里
请按「现象 → 先查什么 → 再查什么」的顺序组织。
```

### 方向 9:发布更新流程

> 对应总计划:第 6 阶段发布更新练习。

```txt
我已经部署了项目。现在我要分别更新前端、后端、数据库结构。
请给我标准操作流程和命令:
1. 改了前端代码后,如何重新 build 前端镜像并平滑替换容器(保留其他服务)
2. 改了后端代码后,如何重新 build server 镜像
3. 数据库新增字段时,如何进 MySQL 容器执行 SQL 并确认 volume 数据还在
4. 更新过程中如何做到不影响已有数据、能快速回滚
请给出每一步的命令和验证方法。
```

---

### 方向 10:最终验收自检

> 对应总计划:第十章验收标准 10 条。

```txt
我的 DeployLab 项目已经部署到服务器。请对照下面 10 条验收标准,给我一组验证命令,
让我能逐条自检:
1. docker compose up -d 能启动
2. 浏览器访问 IP 能看到前端
3. 前端能调 /api/tasks
4. 后端能读写 MySQL
5. 重启容器数据不丢
6. 只有 80 对外开放
7. server/mysql 不暴露公网
8. docker compose logs 能看日志
9. 改前端能重新发布
10. 改后端能重新发布
每条给出:执行什么命令、看到什么结果算通过。
```

## 使用建议

```txt
1. 按顺序用:方向 1~4 是部署前必做,5~7 是部署中,8~10 是部署后。不要跳。
2. 每次只贴一条:AI 拿到「一条提示词 + 你在服务器上跑出来的输出」,
   能给出最准的判断。
3. 结果回写:每个方向跑完后,把关键结论记到 docs/0002-信息资料.md 里
   (比如最终端口方案、目录路径、MySQL 密码),下次直接看。
4. 可复用:这些提示词不只适用于 hp-linux,把「Ubuntu 24.04」「Docker」
   这些字段替换掉,换台机器一样能用。
```
