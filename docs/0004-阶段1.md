# DeployLab 阶段一:本地开发跑通 · 执行方案

> 对应 `0001-总计划.md` 第九章「第 1 阶段:本地开发跑通」。
> 一句话目标:**Win11 本地跑 Vue + Express,数据库用 hp-linux 上新起的 MySQL 容器,把最小读写闭环跑通**。先不碰 Nginx、Redis、docker compose 编排。

---

## 〇、阶段定位与边界

### 0.1 这一阶段的部署形态(与原方案的关键差异)

```txt
Win11 本地开发机                          hp-linux server (Ubuntu 24.04)
┌──────────────────────┐                 ┌─────────────────────────┐
│ Vue dev server :5173 │                 │ (停掉旧的 MySQL/Redis)  │
│ Express 进程   :3000 │ ──Tailscale──▶ │ deploy-mysql 容器 :3306 │
└──────────────────────┘   100.98.102.107 └─────────────────────────┘
```

请求流向:

```txt
浏览器 :5173 ──(Vite proxy /api)──▶ Express :3000 ──Tailscale──▶ hp-linux MySQL :3306
```

### 0.2 与原方案的不同(为什么要这样调)

| 维度 | 原方案 | 本方案 |
|------|--------|--------|
| MySQL 容器在哪 | Win11 本地 Docker Desktop | **hp-linux 上** |
| 连库走什么 | `127.0.0.1:13306` | **Tailscale IP `100.98.102.107:3306`** |
| hp-linux 旧容器 | 不动 | **全部停掉,起新的** |

这样做的好处:**数据库从一开始就在它最终的归宿(hp-linux)上**,省得后面迁移;第 1 阶段就把「Tailscale 组网 + 远程数据库连接」这条链路打通,后面阶段只是把后端也搬过去。

代价:**开发时依赖 Tailscale 连通**(Tailscale 抖一下,本地就连不上库)。能接受就走。

### 0.3 这一阶段「不做什么」(防止跑偏)

| 不做的事 | 留到哪个阶段 |
|---------|-------------|
| Nginx | 第 5 阶段 |
| Redis、`/api/stats` 访问计数 | 第 4 阶段(也在 hp-linux 起新容器) |
| docker compose 多容器编排 | 第 3 阶段 |
| 后端打包成镜像(Dockerfile) | 第 2 阶段 |
| 只对外暴露 80 端口等安全加固 | 第 5 阶段 |

> 第 1 阶段就是「写代码 + 一台远程 MySQL 容器 + 浏览器能增删改查」,心智越简单越好。
> ⚠️ MySQL 端口第 1 阶段会对局域网开放(见第三章),这是**开发期临时方案**,第 5 阶段会收紧。

## 三、在 hp-linux 上准备 MySQL 容器

> ⚠️ 本章全部在 **hp-linux 上**操作(SSH 进去之后)。代码和 SQL 文件先在 Win11 写好,再用 scp 传过去(或直接在 hp-linux 用 vim/nano 创建)。

### 3.1 第一步:停掉 hp-linux 上现有的容器

先看清楚有什么,记录下来(以防以后要恢复),再停:

```bash
# 在 hp-linux 上
docker ps                            # 记下当前容器名、镜像、端口,存档备查
docker ps > ~/existing-containers-$(date +%F).txt

# 按实际容器名停掉(下面是示例名,以你 docker ps 看到的为准)
docker stop <mysql容器名> <redis容器名> <web容器名>
docker ps -a                         # 确认都已 Exited
```

> 这里**只 stop,先不 rm**。第 1 阶段确认新方案跑通前,保留旧容器(占点磁盘)作为回退后路。等整个项目验收通过(总计划第十章)再统一清理 `docker rm`。

### 3.2 第二步:确认 3306 端口已腾出

```bash
# 在 hp-linux 上,确认 3306 没人监听了
ss -ltnp | grep 3306      # 应该没有输出
```

成功标志:无输出(端口空了)。如果还有,说明有非 Docker 的 MySQL 或没停干净,先排查。

### 3.3 第三步:准备建库建表 SQL

在 **Win11** 新建 `server/db/init.sql`:

```sql
CREATE DATABASE IF NOT EXISTS deploy_lab
  CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

USE deploy_lab;

CREATE TABLE IF NOT EXISTS tasks (
  id         INT PRIMARY KEY AUTO_INCREMENT,
  title      VARCHAR(255) NOT NULL,
  status     VARCHAR(50)  NOT NULL DEFAULT 'todo',
  created_at TIMESTAMP    DEFAULT CURRENT_TIMESTAMP
);
```

这就是总计划第六章那张表。把它传到 hp-linux:

```bash
# 在 Win11 Git Bash,仓库根目录
ssh hp-linux-ts "mkdir -p ~/deploy-lab/db"
scp server/db/init.sql hp-linux-ts:~/deploy-lab/db/init.sql
```

### 3.4 第四步:启动新的 MySQL 容器

在 **hp-linux** 上执行:

```bash
docker run -d \
  --name deploy-mysql \
  --restart unless-stopped \
  -e MYSQL_ROOT_PASSWORD=123456 \
  -e MYSQL_DATABASE=deploy_lab \
  -p 3306:3306 \
  -v deploy_mysql_data:/var/lib/mysql \
  -v /home/vichel/deploy-lab/db/init.sql:/docker-entrypoint-initdb.d/init.sql:ro \
  mysql:8.0
```

参数解读(第 1 阶段必须看懂):

```txt
-p 3306:3306          绑 0.0.0.0:3306 → 容器 3306
                      Win11 通过 Tailscale IP 100.98.102.107:3306 连进来
                      ⚠️ 这会对局域网也开放,第 5 阶段会收紧(见 3.6)
-v deploy_mysql_data:/var/lib/mysql        数据持久化(volume)
-v .../init.sql:/docker-entrypoint-initdb.d/init.sql
                                           首次启动自动建库建表
:ro                                        只读挂载
MYSQL_DATABASE=deploy_lab                  顺便建好库
```

> 路径 `/home/vichel/deploy-lab/...` 按你 hp-linux 的实际用户名和目录调整,用 `echo $HOME` 确认。

### 3.5 第五步:验证 MySQL

```bash
# 在 hp-linux 上
# 1) 容器在跑
docker ps --filter name=deploy-mysql

# 2) 进容器看库和表
docker exec -it deploy-mysql mysql -uroot -p123456 -e "USE deploy_lab; SHOW TABLES;"

# 3) 从 Win11 经 Tailscale 连(这才是 Express 将来的连法,关键验证)
#    回到 Win11 Git Bash:
mysql -h 100.98.102.107 -P 3306 -uroot -p123456 -e "SELECT 1"
```

成功标志:
- hp-linux 上 `SHOW TABLES;` 能看到 `tasks`;
- **Win11 能用 Tailscale IP 连上**(第 3 条)——这条通了,Express 才能连。

> Win11 没装 mysql 客户端也没关系,第 3 条可以等第四章用 Express 的 `/api/health` 来替代验证。
>
> ⚠️ **首次启动要点**:MySQL 镜像只在**第一次**启动(即 volume 为空)时执行 `init.sql`。改了 `init.sql` 要 `docker rm -f deploy-mysql && docker volume rm deploy_mysql_data` 后重跑(数据会丢)。

### 3.6 关于端口安全的说明(第 5 阶段再动手)

第 1 阶段 `-p 3306:3306` 会绑到 `0.0.0.0`,意味着同局域网也能扫到。这违反总计划验收第 7 条「server/mysql 不暴露公网」。**这里是有意为之的临时方案**,理由:

- 第 1 阶段聚焦「跑通读写闭环」,不想被防火墙规则分散注意力;
- hp-linux 在内网/Tailscale 后,风险可控;
- **第 5 阶段**会按 `0003-方向7` 收紧:要么改绑 `100.98.102.107:3306:3306`(只 Tailscale 可达),要么后端容器化后直接撤掉外部端口映射、只走 compose 内网。

心里有数即可,本阶段不动。

**`server/src/routes/tasks.js`** —— 任务 CRUD:

```js
const express = require('express');
const router = express.Router();
const db = require('../db');

const VALID_STATUS = ['todo', 'doing', 'done'];

// GET /api/tasks —— 列表
router.get('/', async (_req, res) => {
  const [rows] = await db.query('SELECT * FROM tasks ORDER BY id DESC');
  res.json(rows);
});

// POST /api/tasks —— 新增
router.post('/', async (req, res) => {
  const { title } = req.body;
  if (!title || !title.trim()) {
    return res.status(400).json({ error: 'title 不能为空' });
  }
  const [r] = await db.query(
    'INSERT INTO tasks (title) VALUES (?)',
    [title.trim()]
  );
  res.status(201).json({ id: r.insertId, title: title.trim(), status: 'todo' });
});

// PATCH /api/tasks/:id/status —— 改状态
router.patch('/:id/status', async (req, res) => {
  const { status } = req.body;
  if (!VALID_STATUS.includes(status)) {
    return res.status(400).json({ error: `status 必须是 ${VALID_STATUS.join('/')}` });
  }
  const [r] = await db.query(
    'UPDATE tasks SET status=? WHERE id=?',
    [status, req.params.id]
  );
  if (r.affectedRows === 0) return res.status(404).json({ error: 'task 不存在' });
  res.json({ id: Number(req.params.id), status });
});

// DELETE /api/tasks/:id —— 删除
router.delete('/:id', async (req, res) => {
  await db.query('DELETE FROM tasks WHERE id=?', [req.params.id]);
  res.status(204).end();
});

module.exports = router;
```

### 5.2 Vite proxy 配置(关键)

编辑 `frontend/vite.config.js`,加 `/api` 代理,前端用相对路径就能打到后端,和将来 Nginx 路径完全一致:

```js
import { defineConfig } from 'vite';
import vue from '@vitejs/plugin-vue';

export default defineConfig({
  plugins: [vue()],
  server: {
    port: 5173,
    proxy: {
      '/api': {
        target: 'http://localhost:3000',
        changeOrigin: true,
      },
    },
  },
});
```

> 这个 proxy 就是第 5 阶段 Nginx `proxy_pass /api` 的「开发版」。前端只认 `/api/...`,后端在哪、MySQL 在哪,前端完全不关心。

### 5.3 页面实现

第 1 阶段不拆组件、不引路由库、不引 axios,全部塞进 `frontend/src/App.vue`,用顶部 tab 切三个视图。把 `App.vue` 替换成:

```vue
<script setup>
import { ref, onMounted } from 'vue';

const API = (p, opt) => fetch('/api' + p, opt).then(r =>
  r.status === 204 ? null : r.json()
);

// ---- 任务列表 ----
const tasks = ref([]);
const loading = ref(false);
async function loadTasks() {
  loading.value = true;
  tasks.value = await API('/tasks');
  loading.value = false;
}

// ---- 新增 ----
const newTitle = ref('');
async function addTask() {
  if (!newTitle.value.trim()) return;
  await API('/tasks', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ title: newTitle.value }),
  });
  newTitle.value = '';
  await loadTasks();
}

// ---- 改状态 / 删除 ----
async function setStatus(id, status) {
  await API(`/tasks/${id}/status`, {
    method: 'PATCH',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ status }),
  });
  await loadTasks();
}
async function removeTask(id) {
  await API(`/tasks/${id}`, { method: 'DELETE' });
  await loadTasks();
}

// ---- 系统状态 ----
const tab = ref('list');
const health = ref(null);
async function loadHealth() {
  try {
    health.value = await API('/health');
  } catch {
    health.value = { status: 'error' };
  }
}

onMounted(() => {
  loadTasks();
  loadHealth();
});
</script>
```

## 八、常见坑速查表

| 现象 | 大概率原因 | 怎么查 / 怎么修 |
|------|-----------|----------------|
| 后端启动报 `ECONNREFUSED 100.98.102.107:3306` | Tailscale 没连 / MySQL 容器没起 / 端口没绑 | Win11 `ping 100.98.102.107`;hp-linux `docker ps`、`ss -ltnp \| grep 3306` |
| 后端报 `ETIMEDOUT` 连 100.98.102.107 | Tailscale 抖动或没登录同账号 | Win11 `tailscale status` 看 hp-linux 是否 active |
| 后端报 `ER_ACCESS_DENIED_ERROR` | 密码错 / 库名错 | 核对 `.env` 的 `DB_PASSWORD`、`DB_NAME` 与 `docker run` 的 `-e` 一致 |
| hp-linux `docker run` 报 `bind: address already in use` | 3306 还被占(旧容器没停干净) | `docker ps -a` 看残留;`ss -ltnp \| grep 3306` 找进程 |
| 改了 `init.sql` 没生效 | MySQL 只在首次初始化执行 | `docker rm -f deploy-mysql && docker volume rm deploy_mysql_data` 后重跑 |
| 前端 fetch `/api/tasks` 报 CORS 或 404 | Vite proxy 没配 / 没重启 Vite | 改完 `vite.config.js` 要重启 `npm run dev` |
| 前端请求打到 `localhost:3000` 而非走代理 | fetch 写了绝对 URL | 前端一律用相对路径 `/api/...`,不要带 `http://localhost:3000` |
| 容器起了但 Win11 死活连不上 | hp-linux 防火墙拦了 / 端口绑到 127.0.0.1 | `docker port deploy-mysql` 看绑定;`-p 3306:3306` 应是 `0.0.0.0:3306` |
