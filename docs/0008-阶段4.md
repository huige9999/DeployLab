# DeployLab 阶段四：加入 Redis · 执行方案

> 对应 `0001-总计划.md` 第九章「第 4 阶段：加入 Redis」。
> 一句话目标：**在 Compose 里再加一个 `redis` 服务，让 server 连上它，用 Redis 实现一个「API 访问计数」——`GET /api/stats` 返回访问次数，`/api/health` 真正探测 Redis 存活（不再是 `pending`）。**
> 这一阶段没有迁移、没有数据丢失风险：Redis 是全新的无状态缓存容器，加进来就行。
> 和阶段 3 同理，最重要的心智是：**server 里 `REDIS_HOST` 写 Compose 服务名 `redis`，而不是 `localhost`**——和 `DB_HOST=mysql` 是同一堂课。

---

## 〇、阶段定位与边界

### 0.1 这一阶段的部署形态

阶段 3 有 `mysql` + `server` 两个服务；阶段 4 在同一份 `docker-compose.yml` 里**多加一个 `redis` 服务**，server 同时连两个：

```txt
Win11 开发机                              hp-linux server (Ubuntu 24.04)
┌───────────────────────┐                ┌──────────────────────────────────────────────┐
│ Vue dev server  :5173 │                │  Compose 项目:deploy-lab                      │
│  └ Vite proxy /api    │ ─Tailscale──▶ │  默认网络:deploy-lab_default(阶段3 已建)      │
│     → 100.98.102.107  │  100.98.102.107│  ┌──────────┐   ┌─────────┐  ┌──────────┐   │
│        :3000          │                │  │  server  │──▶│  mysql  │  │  redis   │   │
└───────────────────────┘                │  │ Express  │──▶│  :3306  │  │  :6379   │   │
                                         │  │  :3000   │   └─────────┘  └──────────┘   │
                                         │  └──────────┘        ▲            ▲          │
                                         │   -p 3000:3000       │            │          │
                                         │   DB_HOST=mysql      └────────────┘          │
                                         │   REDIS_HOST=redis   server 同时连两个        │
                                         └──────────────────────────────────────────────┘
```

请求流向（与阶段 3 相比，**跨机段、到 mysql 的链路完全不变**，只是 server 多了一条到 redis 的连接）：

```txt
浏览器 :5173 ─(Vite proxy /api)─▶ server:3000 ┬─(Compose 网络 DNS)─▶ mysql:3306   【读写任务】
   │ Win11                          │ hp-linux │                                       
   └── Tailscale 100.98.102.107:3000┘          └─(Compose 网络 DNS)─▶ redis:6379   【计数 + ping】
```

### 0.2 与第 3 阶段的关键差异

| 维度 | 第 3 阶段 | 第 4 阶段（本方案） |
|------|-----------|--------------------|
| Compose 服务数量 | 2（mysql、server） | **3（mysql、server、redis）** |
| server 连接的外部依赖 | 1 个（MySQL） | **2 个（MySQL + Redis）** |
| `health` 里 `redis` 字段 | 写死 `'pending'`（占位） | **真 ping Redis，返回 `ok` / `down`** |
| 是否有 `/api/stats` | 没有 | **有，返回 Redis 里的访问计数** |
| server 新依赖 | mysql2、express、cors、dotenv | **+ ioredis（Redis 客户端）** |
| 数据持久化 | MySQL 走 volume（必须保留） | **Redis 计数默认随容器重启清零（可接受，见 0.4）** |
| 对外端口 | 3000（server）、3306（mysql）临时外露 | **同上；redis 的 6379 不外露**（见 3.1） |

本阶段是阶段 3 的「自然延伸」——Compose 的好处开始显现：**再加一个 service、给 server 加一行环境变量，不用记任何 `docker run`。**

### 0.3 这一阶段「不做什么」（防止跑偏）

| 不做的事 | 留到哪个阶段 |
|---------|-------------|
| 前端容器化 + Nginx 反代、撤掉 3000/3306 外部映射 | 第 5 阶段 |
| 分服务 build / 滚动更新练习 | 第 6 阶段 |
| 给 Redis 加密码、加 volume 持久化（生产化） | 见 7.5，本阶段可选，不强制 |
| 用 Redis 缓存任务列表（真正的「缓存」业务） | 本项目不做，计数够练了 |

> 第 4 阶段就是「Compose 加 redis、server 连 redis、实现计数」三件事。心智越简单越好。

### 0.4 Redis 的计数会丢吗？

**默认会丢。** Redis 默认把数据放内存，容器 `restart` / `down` 后，`api:request_count` 会从 0 重新开始。

这**不是 bug**：

- 这个计数器只是用来练「容器间通信 + Redis 基本用法」，丢了无所谓；
- 真正需要持久化的数据（任务表）在 **MySQL 的 volume** 里，本阶段完全不动它。

如果想让计数也持久化（练 Redis 的 `appendonly` / RDB），看 7.5「（可选）给 Redis 加 volume」。**默认不做，先跑通再说。**

### 0.5 ⚠️ 再说一遍：`REDIS_HOST` 为什么写 `redis`？

和阶段 3 `DB_HOST=mysql` 完全同理：

```yaml
services:
  redis:                          # ← 这个字符串就是 Docker 内部 DNS 名
    image: redis:7-alpine
  server:
    environment:
      REDIS_HOST: redis           # ← 必须和上面的服务名一致
      REDIS_PORT: "6379"
```

> **写「对方在 Compose 内部网络里的名字」，别写 `localhost`。** 写成 `localhost`，server 容器里根本没有 Redis，会直接 `ECONNREFUSED 127.0.0.1:6379`——和阶段 2/3 里写错 `DB_HOST` 是同一个错误。

### 2.1 加 Redis 客户端依赖 `ioredis`

在 Win11 仓库根目录：

```bash
cd server
npm install ioredis
```

> 为什么选 `ioredis`？它是 Node 生态里最主流的 Redis 客户端之一，原生支持 Promise、自动重连、API 简洁（`await client.incr('key')`）。
>
> 装完后 `server/package.json` 的 `dependencies` 会多一行：
> ```json
> "ioredis": "^5.x.x"
> ```
> `package-lock.json` 也会更新——**两个文件都要一起 scp 到 hp-linux**，因为镜像里跑的是 `npm ci`，锁文件必须和 package.json 对得上。

### 2.2 新建 `server/src/redis.js`（对标 `db.js`）

```js
const Redis = require('ioredis');
require('dotenv').config();

const redis = new Redis({
  host: process.env.REDIS_HOST || '127.0.0.1',
  port: Number(process.env.REDIS_PORT) || 6379,
  // 连不上别把整个 server 拖崩：让 ioredis 自己静默重连
  retryStrategy: (times) => Math.min(times * 500, 3000),
  // 学习环境 Redis 无密码；以后加 requirepass 再补 password 字段
  maxRetriesPerRequest: 1,
});

redis.on('connect', () => console.log('[redis] connected'));
redis.on('error', (err) => console.error('[redis] error:', err.message));

module.exports = redis;
```

逐行理解（对标 `db.js` 的 `mysql.createPool`）：

```txt
new Redis({ host, port })
  → 连到 REDIS_HOST:REDIS_PORT。host 走环境变量，默认 127.0.0.1（仅本地裸跑时兜底）

retryStrategy
  → 连断后多久重试一次。这里 500ms 起步、封顶 3s，避免疯狂重连刷日志

maxRetriesPerRequest: 1
  → 某个请求如果 Redis 正好断了，最多重试 1 次就抛错；不让请求无限挂着

on('connect' / 'error')
  → 连接事件打日志，排错第一入口（docker compose logs server 能看到）
```

> ⚠️ 关键：**容器里 `REDIS_HOST` 必须是 Compose 服务名 `redis`**（见 0.5）。这里只是读环境变量，真正写在哪——在 `docker-compose.yml` 里。

### 2.3 新建 `server/src/routes/stats.js`

```js
const express = require('express');
const router = express.Router();
const redis = require('../redis');

// GET /api/stats —— 返回 Redis 里的访问计数
router.get('/', async (_req, res) => {
  try {
    // INCR 写入/读取二合一：key 不存在时建为 0 再 +1
    // 这里只读不写，用 GET
    const count = await redis.get('api:request_count');
    res.json({
      request_count: Number(count) || 0,
      source: 'redis',
    });
  } catch (e) {
    res.status(500).json({ status: 'error', error: e.message });
  }
});

module.exports = router;
```

> 设计说明：
> - 计数的 key 用 `api:request_count`（带命名空间，养成习惯，别用裸 `count`）。
> - `/api/stats` **只读**不写——写入由 2.4 的「计数中间件」统一做，职责分离。
> - Redis 挂了就返回 500，前端状态页能据此显示「Redis 异常」。

### 2.4 改 `server/src/app.js`：挂计数中间件 + stats 路由

完整新内容（在阶段 3 基础上加 2 段）：

```js
require('dotenv').config();
const express = require('express');
const cors = require('cors');

const redis = require('./redis');
const healthRouter = require('./routes/health');
const tasksRouter = require('./routes/tasks');
const statsRouter = require('./routes/stats');

const app = express();
app.use(cors());
app.use(express.json());

// ⭐ 访问计数中间件：每个 /api 请求都让 Redis 计数 +1
// 放在所有业务路由之前，这样 health / tasks / stats 都会被计入
app.use('/api', async (_req, _res, next) => {
  try {
    await redis.incr('api:request_count');
  } catch (e) {
    // 计数失败不能影响业务请求，吞掉、继续
    console.error('[redis] incr failed:', e.message);
  }
  next();
});

app.use('/api', healthRouter);
app.use('/api/tasks', tasksRouter);
app.use('/api/stats', statsRouter);

const PORT = process.env.PORT || 3000;

app.listen(PORT, () => {
  console.log(`Server is running on port ${PORT}`);
});
```

逐段理解：

```txt
app.use('/api', async (_req, _res, next) => { await redis.incr(...); next(); })
  → 每个以 /api 开头的请求，进业务逻辑前先把 Redis 计数 +1
  → INCR 是原子操作，并发请求不会漏数、不会重复
  → try/catch 包住：Redis 挂了只是不计数，不能让用户请求失败

app.use('/api/stats', statsRouter)
  → 新路由，挂载路径 /api/stats，对应总计划第五章的接口

redis = require('./redis') 提前到顶部
  → 模块单例：整个进程共用一个 Redis 连接，避免每个请求都新建连接
```

> 顺序很重要：**计数中间件必须在业务路由之前**。Express 是按注册顺序匹配的，先到中间件（计数），再 `next()` 进 health/tasks/stats。

### 2.5 改 `server/src/routes/health.js`：真实 ping Redis

把原来写死的 `redis: 'pending'` 换成真的探测：

```js
const express = require('express');
const router = express.Router();
const db = require('../db');
const redis = require('../redis');

router.get('/health', async (_req, res) => {
  // 并发探测两个依赖，谁挂了不影响另一个的判断
  const checks = await Promise.allSettled([
    db.query('SELECT 1'),
    redis.ping(),
  ]);

  const dbOk = checks[0].status === 'fulfilled';
  const redisOk = checks[1].status === 'fulfilled';

  res.json({
    status: dbOk && redisOk ? 'ok' : 'degraded',
    db: dbOk ? 'ok' : 'down',
    redis: redisOk ? 'ok' : 'down',      // ⭐ 从 'pending' 升级为真实状态
    version: process.env.APP_VERSION || 'unknown',
  });
});

module.exports = router;
```

理解：

```txt
Promise.allSettled([db.query, redis.ping])
  → 两个探测「同时」发、互不阻塞；任何一个失败都不会让另一个抛异常

redis.ping() → Redis 回 'PONG' 即视为 ok
  → 这是探测 Redis 存活的标准做法，比 try SET 更轻

status: 'degraded'
  → 只要有一个依赖 down，整体就不算 ok，但也不算彻底 error（前端能区分显示）
```

> 比阶段 3 多了一个 `degraded` 中间态——这样前端状态页能精确告诉你「后端活着，但 Redis 挂了」，对排错非常有用。

## 三、改 `docker-compose.yml`（Win11 本地）

> 只加 1 个 service（redis）、给 server 加 2 个环境变量 + 1 个 depends。MySQL 部分一个字都不改。

### 3.1 在阶段 3 基础上增量改动

```yaml
name: deploy-lab

services:
  mysql:
    image: mysql:8.0
    restart: unless-stopped
    environment:
      MYSQL_ROOT_PASSWORD: "123456"
      MYSQL_DATABASE: deploy_lab
    ports:
      - "3306:3306"                    # ⚠️ 临时对外，第 5 阶段撤掉
    volumes:
      - deploy_mysql_data:/var/lib/mysql
      - ./server/db/init.sql:/docker-entrypoint-initdb.d/init.sql:ro
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "127.0.0.1", "-uroot", "-p123456"]
      interval: 5s
      timeout: 5s
      retries: 10
      start_period: 30s

  # ⭐ 新增：Redis 服务
  redis:
    image: redis:7-alpine
    restart: unless-stopped
    # ⚠️ 故意不写 ports —— 6379 只在 Compose 内网暴露，不映射到宿主机
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 10
      start_period: 5s

  server:
    build:
      context: ./server
      dockerfile: DockerFile
    image: deploy-lab-server:v2         # ⭐ v1 → v2，标记「接了 Redis」的镜像
    restart: unless-stopped
    depends_on:
      mysql:
        condition: service_healthy
      redis:
        condition: service_healthy      # ⭐ server 等 mysql 和 redis 都健康再起
    ports:
      - "3000:3000"                     # ⚠️ 临时对外，第 5 阶段撤掉
    environment:
      PORT: "3000"
      DB_HOST: mysql
      DB_PORT: "3306"
      DB_USER: root
      DB_PASSWORD: "123456"
      DB_NAME: deploy_lab
      REDIS_HOST: redis                 # ⭐ Compose 服务名，不是 localhost
      REDIS_PORT: "6379"
      APP_VERSION: "1.0.0"

volumes:
  deploy_mysql_data:
    external: true
```

### 3.2 逐段理解（只看新增部分）

```txt
services.redis
  → 新的 service。服务名 redis 即内部 DNS 名，server 用 REDIS_HOST=redis 连它

image: redis:7-alpine
  → Redis 7 官方 alpine 版，体积小。不带密码（学习环境，见 7.5）

redis.healthcheck: redis-cli ping
  → 用 redis-cli 探活，配合 server 的 condition: service_healthy
  → 这样 server 起来时 Redis 已 ready，首启不会 ECONNREFUSED

server.environment.REDIS_HOST / REDIS_PORT
  → server 读这两个变量去连 redis:6379（见 2.2 的 redis.js）

server.depends_on.redis.condition: service_healthy
  → server 等 redis 的 healthcheck 通过再启动（和 mysql 同等待法）

server.image: deploy-lab-server:v2
  → 改代码了换版本号，方便回滚：万一 v2 有问题，docker compose 能指回 v1
```

### 3.3 ⚠️ 为什么 Redis 不映射端口？

```txt
ports 段没写 redis → 6379 只存在于 deploy-lab_default 内网
  → 只有同网络的 server 容器能连 redis:6379
  → 宿主机、外网都连不到 → 更安全
```

对比：mysql、server 现在都临时映射了端口（`3306:3306`、`3000:3000`），那是为了**阶段 4 前方便 Win11 跨机调试**。Redis 没有这个需求（前端不直连 Redis），所以从一开始就**不外露**——这正好预演第 5 阶段「只暴露 80」的安全收口。

> 排错时如果要在宿主机连 Redis，用 `docker compose exec redis redis-cli`（见 8.2），不需要映射端口。

### 3.4 Redis 要不要 volume？

**本阶段：不挂。** Redis 计数随容器重启清零，可接受（见 0.4）。要持久化看 7.5，作为可选增强。

### 3.5 关于 Redis 密码

本阶段 Redis 无密码（默认），和阶段 3 的 MySQL 弱密码一样，是**学习环境简化**。生产化（`requirepass` + `redis.js` 里补 `password` 字段）留到以后，本方案不展开。

## 七、核心知识点（搞懂就不怕加新依赖）

### 7.1 Redis 和 MySQL 在 Compose 里是「对称」的

| 维度 | MySQL | Redis |
|------|-------|-------|
| 服务名（= DNS 名） | `mysql` | `redis` |
| server 连它用 | `DB_HOST=mysql` | `REDIS_HOST=redis` |
| 端口 | 3306 | 6379 |
| healthcheck 命令 | `mysqladmin ping` | `redis-cli ping` |
| 数据持久化 | volume（必须） | 默认内存（可选 volume） |
| 客户端库 | `mysql2` | `ioredis` |
| 客户端模块 | `db.js`（pool） | `redis.js`（单例连接） |

**理解了这套「模板」，以后加任何依赖（比如 Elasticsearch）都是照葫芦画瓢：加 service + 加环境变量 + 写客户端模块 + 路由里用。**

### 7.2 `INCR` 为什么适合做计数器？

```txt
INCR api:request_count
  → 原子操作：key 不存在先建为 0，再 +1，返回新值
  → 并发请求不会互相覆盖、不会丢数
  → 比应用层「先 GET 再 +1 再 SET」安全得多（后者在并发下会漏数）
```

本方案用中间件在每个 `/api` 请求调一次 `INCR`，天然支持并发。

### 7.3 为什么用一个共享的 Redis 连接（模块单例）？

```js
// redis.js：module.exports = redis  → 整个进程共用一个连接
// 而不是每个请求都 new Redis()
```

```txt
每次请求 new Redis() → 频繁建连/断连 → 慢、还会打爆 Redis 的最大连接数
模块级单例           → 一个连接复用，ioredis 内部还能 pipeline 提升吞吐
```

`db.js` 的 `mysql.createPool` 也是同一个思想：连接池复用，避免每请求建连。

### 7.4 Redis 挂了，server 为什么不能跟着挂？

看 `redis.js` 的 `retryStrategy` + `health.js` 的 `Promise.allSettled`：

```txt
Redis 挂 → ioredis 静默重连（不打断 server）
        → health 里 redis.ping() 失败 → redis: 'down'，status: 'degraded'
        → tasks 接口照常工作（它只用 MySQL，不碰 Redis）
        → 只有 /api/stats 受影响（返回 500）
```

> 这是「依赖降级」的设计：缓存/计数挂了，核心业务（任务增删改查）还能用。生产环境对 Redis 这类「可降级依赖」常这么处理；对 MySQL 这类「硬依赖」则必须报错。

### 7.5 （可选）给 Redis 加持久化（练 volume）

如果想体验「Redis 容器重启后计数还在」，给 redis 服务加 volume：

```yaml
  redis:
    image: redis:7-alpine
    restart: unless-stopped
    command: ["redis-server", "--appendonly", "yes"]   # 开 AOF 持久化
    volumes:
      - deploy_redis_data:/data                         # 持久化目录
```

并在文件末尾 `volumes:` 段加：

```yaml
volumes:
  deploy_mysql_data:
    external: true
  deploy_redis_data:    # ⭐ 新增：compose 自动创建并管理（非 external）
```

效果：`docker compose down`（不加 `-v`）后再 `up`，计数还在。

> 本阶段**默认不开**——计数器丢了无所谓，先把「连接 + 计数」跑通。练 volume 用 8.3 的对照实验即可。

### 8.1 看三个服务都在同一网络

```bash
docker network inspect deploy-lab_default
# "Containers" 段：应同时有 mysql、redis、server 三个成员
```

### 8.2 进 Redis 交互式命令行

```bash
docker compose exec redis redis-cli
# 进去后：
127.0.0.1:6379> GET api:request_count
127.0.0.1:6379> INCR api:request_count      # 手动 +1，看 /api/stats 是否跟着变
127.0.0.1:6379> KEYS *                       # 看所有 key
127.0.0.1:6379> FLUSHALL                      # 清空（慎用，计数归零）
127.0.0.1:6379> exit
```

> **不映射端口也能进 Redis**——靠 `docker compose exec`，这正是 3.3「不暴露端口」的安全模型。

### 8.3 对照实验：停掉 Redis，看 health 降级

```bash
docker compose stop redis
curl -s http://localhost:3000/api/health
# 期望:{"status":"degraded","db":"ok","redis":"down",...}
curl -s http://localhost:3000/api/tasks | head -c 100
# 期望:任务列表照常返回（MySQL 没受影响）—— 核心 7.4 的降级设计
curl -s http://localhost:3000/api/stats
# 期望:500（stats 依赖 Redis）

docker compose start redis
sleep 2
curl -s http://localhost:3000/api/health      # 恢复 redis:ok
```

> 这个实验最能体现阶段 4 的设计取舍：**Redis 是可降级依赖**，挂了不拖垮任务功能。

### 8.4 对照实验：故意写错 `REDIS_HOST`

临时把 compose 里 `REDIS_HOST` 改成 `localhost`：

```bash
docker compose up -d server          # 应用环境变量变更
docker compose logs --tail 20 server
# 期望:[redis] error: connect ECONNREFUSED 127.0.0.1:6379
curl -s http://localhost:3000/api/health    # redis:down
```

改回 `redis` 再 `docker compose up -d server`——再次体会「服务名 vs localhost」这堂贯穿阶段 2/3/4 的课。

### 8.5 计数随 Redis 重启清零（验证默认不持久）

```bash
curl -s http://localhost:3000/api/stats    # 记下计数，比如 42
docker compose restart redis
sleep 2
curl -s http://localhost:3000/api/stats    # 期望回到小数字（重启期间 server 自己发的探测也算）
```

> 默认配置下 Redis 数据在内存，`restart` 后计数重置。想让它不丢，开 7.5。

## 九、常见坑速查表

| 现象 | 大概率原因 | 怎么查 / 怎么修 |
|------|-----------|----------------|
| `up` 后 server 日志 `Cannot find module 'ioredis'` | 镜像没重新 build，容器里没装 ioredis | `docker compose build server` 后再 `docker compose up -d server` |
| `[redis] error: ENOTFOUND redis` | `REDIS_HOST` 写错 / 服务名和 yml 不一致 | 核对 `services.redis` 与 `REDIS_HOST=redis`；`docker compose exec server ping redis` |
| `[redis] error: ECONNREFUSED 127.0.0.1:6379` | `REDIS_HOST=localhost` | 改成 `redis` 后 `docker compose up -d server` |
| health 返回 `redis:down`，但 server 起来了 | redis 服务没起 / healthcheck 没过 | `docker compose ps` 看 redis 状态；`docker compose logs redis` |
| `/api/stats` 返回 500 | Redis 连不上 | 同上两行；`docker compose exec server wget -qO- redis:6379` 探连通性 |
| `npm ci` 报锁文件不一致 | 只传了 package.json，没传 package-lock.json | 重新 `scp server/package-lock.json`，再 build |
| 计数永远不变（卡在某个值） | 计数中间件没生效（挂在路由之后） | 确认 `app.use('/api', ...计数中间件...)` 在 `app.use('/api/tasks'...)` 之前 |
| build 慢 / 重复装依赖 | 没命中 Docker 缓存层 | 确认 Dockerfile 先 `COPY package*.json` 再 `npm ci`（阶段 2 已是对的） |
| Win11 curl 不通 stats | 3000 未映射 / Tailscale 断 | `docker compose ps` 看 PORTS；`tailscale status` |
| 前端状态页 Redis 显示 `...`（不更新） | 没在切 tab 时调 `loadStats()` | 检查 App.vue 里状态 tab 的 `@click` 是否含 `loadStats()` |
